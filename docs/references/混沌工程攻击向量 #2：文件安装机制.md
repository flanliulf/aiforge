# 混沌工程攻击向量 #2：文件安装机制

> **归档分组：A 组 — 当前 MVP 风险**
> 本文涉及的攻击场景直接影响 MVP 阶段的核心功能（文件复制、符号链接、冲突处理），需在 M1 里程碑前完成加固。

文件安装是 aiforge 最终"动手改东西"的环节，前面所有设计再完美，这里出问题用户就会受到实际伤害——文件丢失、配置损坏、工具行为异常。

---

## 先梳理文件安装的完整流程

```
aiforge 的文件安装根据模式不同，有三条路径:

路径 1: 复制模式（默认，无 -l 参数）
  源文件 → cp → 目标位置
  结果: 独立副本，与源无关联

路径 2: 符号链接模式（-l 参数）
  源文件 → symlink → 目标位置
  结果: 目标指向源，改源即改目标

路径 3: flatten 模式（Cursor 的 .cursor/rules/ 场景）
  源目录中多个文件 → 合并/重命名 → 单个目标文件
  结果: 多变一，有内容转换

每条路径有三个阶段:
  阶段 1: 确定源路径和目标路径
  阶段 2: 处理目标位置的现有内容（冲突）
  阶段 3: 执行文件操作
```

开始攻击。

## 💥 攻击场景 2.1：目标位置已有用户手写内容

```
场景:
  用户之前手动创建了 ~/.copilot/agents/api-dev.md
  写了自己的 agent 配置（花了两小时调试 prompt）

  然后运行:
  $ npx aiforge https://gitlab.xxx.com/team/ai-base.git -g

  知识仓库中恰好也有 agents/api-dev.md

  行为:
    aiforge 复制 agents/api-dev.md → ~/.copilot/agents/api-dev.md
    → 用户手写的内容被覆盖
    → 没有任何确认提示
    → 没有备份
    → 两小时的工作丢失

  变体场景:
    用户在知识仓库的基础上做了本地修改（比如加了几行个人偏好）
    下次 aiforge 更新时，本地修改被覆盖

暴露的脆弱点:
  1. 没有冲突检测
  2. 没有覆盖确认
  3. 没有备份机制
  4. 无法区分 "aiforge 上次安装的" 和 "用户手写的"
```

**这是最危险的攻击场景。数据丢失是不可逆的。**

### 加固方案

```
三层防护:

层 1 — 冲突检测:
  在执行安装前，扫描所有目标路径
  if (targetExists && !isAiforgeManaged(target)) {
    // 目标存在，且不是 aiforge 上次安装的
    conflicts.push(target);
  }

层 2 — 冲突处理策略:

> ✅ 已采纳为 ADR-011（修订版）：默认交互式选择，推荐 backup，--force 直接覆盖，--dry-run 仅预览。
  $ npx aiforge ... -g

  ⚠️ 发现 2 个文件冲突:

  ~/.copilot/agents/api-dev.md
    本地文件: 2024-01-15 修改, 1.2KB
    仓库文件: 2024-01-20 修改, 0.8KB

  ~/.copilot/skills/git/
    本地目录: 包含 3 个文件（含本地修改）
    仓库目录: 包含 2 个文件

  如何处理?
    [o] 覆盖全部（overwrite）
    [s] 跳过冲突文件（skip）
    [b] 备份后覆盖（backup）       ← 推荐
    [d] 逐个查看差异（diff）
    [a] 中止安装（abort）

层 3 — 备份机制:
  选择 backup 时:
    ~/.copilot/agents/api-dev.md
      → ~/.copilot/agents/api-dev.md.aiforge-backup-20240120
  
  或者统一备份目录:
    ~/.aiforge/backups/20240120-153022/
      └── .copilot/agents/api-dev.md
```

### "aiforge 管理标记"的设计

```
关键问题: 如何区分"aiforge 安装的文件"和"用户手写的文件"?

方案 A: 文件头标记
  aiforge 安装的文件在头部添加注释:
    <!-- managed by aiforge | source: gitlab.xxx.com/team/ai-base | hash: a1b2c3 -->
    # API Developer Agent
    ...

  检测: 读取文件前几行，查找标记

  优点: 直观，用户能看到
  缺点: 修改了文件内容，可能影响 AI 工具解析
        如果用户编辑文件时删了注释，标记就丢了

方案 B: 元数据清单文件
  ~/.aiforge/manifest.json
  {
    "installations": {
      "~/.copilot/agents/api-dev.md": {
        "source": "gitlab.xxx.com/team/ai-base",
        "sourceHash": "a1b2c3d4",
        "installedAt": "2024-01-20T15:30:22Z",
        "installedHash": "e5f6g7h8"
      },
      "~/.copilot/skills/git/": {
        "source": "gitlab.xxx.com/team/ai-base",
        "sourceHash": "i9j0k1l2",
        "installedAt": "2024-01-20T15:30:22Z",
        "type": "symlink"
      }
    }
  }

  检测: 
    目标文件存在 → 查 manifest → 
      有记录 → 比对 hash → 
        hash 一致: aiforge 安装，用户未修改 → 安全覆盖
        hash 不一致: aiforge 安装，但用户做了修改 → 冲突！
      无记录 → 用户手写的文件 → 冲突！

  优点: 不修改文件内容，不影响 AI 工具
        能检测"用户是否修改过 aiforge 安装的文件"
  缺点: 额外的状态文件需要维护
        manifest 本身可能损坏或丢失

推荐: 方案 B（manifest）

  原因:
    1. 不修改文件内容是硬约束（AI 工具可能对格式敏感）
    2. manifest 丢失的降级策略: 所有文件视为"未知来源"→ 触发冲突确认
       （安全的降级方向: 宁可多问，不要静默覆盖）
    3. manifest 为后续的"增量更新"功能提供了基础设施
```

## 💥 攻击场景 2.2：符号链接的七种死法

```
符号链接模式 (-l) 的核心前提:
  仓库被克隆到本地 → 创建符号链接指向仓库中的文件
  用户后续 git pull 更新仓库 → 符号链接自动指向新内容

死法 1: 源目录被删除/移动
─────────────────────────
  $ npx aiforge ... -g -l
  → 仓库克隆到 /tmp/aiforge-xxxxx/ai-base/
  → 创建 ~/.copilot/agents/ → /tmp/aiforge-xxxxx/ai-base/agents/

  系统重启后 /tmp 被清理
  → 符号链接指向不存在的路径
  → AI 工具: "找不到配置文件"
  → 用户: "昨天还好好的，今天就坏了"

  根因: 克隆到 /tmp 是致命的设计错误

  修复:
    默认克隆位置: ~/.aiforge/repos/<repo-hash>/
    不能用 /tmp，不能用系统会自动清理的目录
    克隆路径写入 manifest.json，方便追踪


死法 2: 符号链接权限问题（Windows）
───────────────────────────────
  Windows 下创建符号链接需要:
    方式 A: 管理员权限
    方式 B: 开发者模式（Windows 10 1703+）
    方式 C: 组策略 SeCreateSymbolicLinkPrivilege

  如果都没有:
    fs.symlinkSync() → Error: EPERM
    → "权限不足" 但用户不知道怎么修

  修复:
    try {
      fs.symlinkSync(source, target);
    } catch (e) {
      if (e.code === 'EPERM' && process.platform === 'win32') {
        warn('Windows 下创建符号链接需要开发者模式或管理员权限');
        warn('');
        warn('解决方案（任选其一）:');
        warn('  1. 设置 → 更新和安全 → 开发者选项 → 开启开发者模式');
        warn('  2. 以管理员身份运行终端');
        warn('  3. 不使用 -l 参数（使用复制模式替代）');
        warn('');
        warn('正在回退到复制模式...');
        fs.copyFileSync(source, target);  // 自动降级
      }
    }


死法 3: 相对路径 vs 绝对路径
──────────────────────────
  创建符号链接时:
    绝对路径: ~/.copilot/agents → /home/user/.aiforge/repos/xxx/agents
    相对路径: ~/.copilot/agents → ../../.aiforge/repos/xxx/agents

  如果用绝对路径:
    → 用户 home 目录改名/迁移后链接断裂
    → 在 CI/CD 环境中路径完全不同（如 /home/runner/）

  如果用相对路径:
    → 计算相对路径的逻辑容易出错
    → 目标目录层级变化后相对路径就错了

  修复:
    全局安装 → 绝对路径（因为源和目标都在 ~ 下，路径稳定）
    项目级安装 → 相对路径（因为项目可能被移动/重命名，
                          但项目内的相对关系不变）

> ✅ 已采纳为 ADR-006（修订版）：全局安装使用绝对路径，项目级安装使用相对路径 + 自动 .gitignore。


死法 4: 循环符号链接
─────────────────
  如果用户之前手动创建了:
    ~/.copilot/agents → ~/.config/ai/agents

  然后 aiforge 尝试:
    在 ~/.copilot/agents/ 下创建链接
    → 实际写入 ~/.config/ai/agents/（因为 ~/.copilot/agents 是链接）
    → 如果 ~/.config/ai/agents 又指向别的地方...

  修复:
    安装前检测目标路径是否是符号链接:
    const realTarget = fs.realpathSync(targetDir);
    if (realTarget !== path.resolve(targetDir)) {
      warn(`目标目录 ${targetDir} 是符号链接，指向 ${realTarget}`);
      warn('aiforge 将在实际目录中安装');
    }


死法 5: 仓库更新后的"幽灵文件"
──────────────────────────────
  时间线:
    T1: 仓库有 agents/api-dev.md、agents/pm.md
        aiforge -l 创建两个符号链接

    T2: 仓库删除了 agents/pm.md
        用户 git pull

    T3: 符号链接 ~/.copilot/agents/pm.md 还在
        但指向的 agents/pm.md 已经不存在了
        → 悬空的符号链接（dangling symlink）
        → ls 能看到文件名但 cat 报错 "No such file"
        → AI 工具可能崩溃或行为异常

  修复:
    npx aiforge sync 命令:
      1. 读取 manifest.json 中的安装记录
      2. 检查每个符号链接：源文件是否存在
      3. 删除悬空链接
      4. 创建新增文件的链接
      5. 报告变更

    或者: 每次 aiforge 运行时自动执行"健康检查"
      function healthCheck() {
        const manifest = loadManifest();
        for (const [target, info] of Object.entries(manifest.installations)) {
          if (info.type === 'symlink') {
            if (!fs.existsSync(fs.readlinkSync(target))) {
              warn(`悬空链接: ${target}`);
              danglingLinks.push(target);
            }
          }
        }
      }


死法 6: 目录链接 vs 文件链接的混淆
────────────────────────────────
  skills 的安装类型是 "directories":
    仓库中: skills/tapd/  (一个目录，含多个文件)
  
  方式 A: 链接整个目录
    ~/.copilot/skills/tapd → ~/.aiforge/repos/xxx/skills/tapd/
    ✅ 简单
    ❌ 如果用户要在本地给 tapd skill 加一个文件（不改仓库），做不到
       因为链接指向的是仓库目录，本地加的文件会出现在仓库中

  方式 B: 链接目录中的每个文件
    ~/.copilot/skills/tapd/SKILL.md → ~/.aiforge/repos/xxx/skills/tapd/SKILL.md
    ~/.copilot/skills/tapd/extra.md → ~/.aiforge/repos/xxx/skills/tapd/extra.md
    ✅ 用户可以在 tapd/ 目录下加自己的文件（不会回写到仓库）
    ❌ 更复杂，文件增减时需要同步

  修复:
    MVP 阶段: 方式 A（链接整个目录）
    原因: 简单，且"往 skill 里加私有文件"是极端边缘场景
    文档说明: "使用 -l 模式时，skill 目录整体链接，本地修改会反映在仓库中"


死法 7: Git 操作与符号链接的交互
─────────────────────────────
  如果项目本身是一个 git 仓库（几乎一定是）:
    my-project/
    ├── .git/
    ├── .github/
    │   └── copilot-instructions.md → ~/.aiforge/repos/xxx/instructions/...
    ├── .cursor/
    │   └── rules/
    │       └── tapd.mdc → ~/.aiforge/repos/xxx/skills/tapd/...
    └── src/

  问题:
    git status 会显示这些符号链接
    git add . 会把符号链接加入版本控制
    团队其他成员 clone 后，符号链接指向的路径不存在
    → "为什么我的项目一堆断链接？"

  修复:
    项目级安装 (-p) 使用符号链接时，自动在 .gitignore 中添加忽略规则
    → 全局安装 (-g) + 符号链接 → 绝对路径
    → 项目级安装 + 符号链接 → 相对路径 + 自动添加 .gitignore 忽略

    if (options.link && options.project) {
      // 使用相对路径创建符号链接
      const relativePath = path.relative(targetDir, sourceDir);
      fs.symlinkSync(relativePath, targetPath);
      // 自动在 .gitignore 中添加忽略规则
      appendToGitignore(projectRoot, targetPath);
      info('项目级符号链接已创建（相对路径），已自动添加到 .gitignore');
    }
    但这改动了用户的 .gitignore，也不理想
```

## 💥 攻击场景 2.3：部分安装失败（半完成状态）

```
场景:
  aiforge 需要安装 5 个文件:
    1. ~/.copilot/agents/api-dev.md      ✅ 成功
    2. ~/.copilot/agents/pm.md           ✅ 成功
    3. ~/.copilot/skills/tapd/           ❌ 失败（磁盘满）
    4. ~/.copilot/skills/git/            ⏸️ 未执行
    5. ~/.copilot/instructions/...       ⏸️ 未执行

  结果:
    系统处于"半安装"状态
    agents 装了但 skills 没装
    用户不清楚哪些装了哪些没装

  更糟的场景:
    如果第 3 步是"覆盖旧文件"失败:
      旧文件已经被删除（准备写入新文件），但新文件写入失败
      → 旧文件丢失，新文件也没有
      → 数据丢失

暴露的脆弱点:
  文件安装不是原子操作
  没有事务语义（要么全成功，要么全回滚）
```

### 加固方案

```
方案: 两阶段安装

阶段 1 — 准备（Prepare）:
  不动任何目标文件，只做:
    - 检查所有源文件是否存在
    - 检查所有目标路径是否可写
    - 检查磁盘空间是否足够
    - 如果有冲突文件，先创建备份
    - 将所有"要做的操作"写入一个执行计划

  执行计划示例:
  [
    { op: 'copy', from: '...agents/api-dev.md', to: '~/.copilot/agents/api-dev.md' },
    { op: 'copy', from: '...agents/pm.md', to: '~/.copilot/agents/pm.md' },
    { op: 'mkdir', path: '~/.copilot/skills/tapd/' },
    { op: 'copy', from: '...skills/tapd/SKILL.md', to: '~/.copilot/skills/tapd/SKILL.md' },
    ...
  ]

  如果准备阶段发现任何问题 → 立即报错，不执行任何操作

阶段 2 — 执行（Execute）:
  按计划逐条执行
  每条操作成功后记入 manifest
  如果某条失败:
    → 已执行的操作记录在 manifest 中（用于后续清理）
    → 报告哪些成功、哪些失败
    → 提供 npx aiforge rollback 命令回滚

代码骨架:

  function install(plan) {
    const executed = [];
  
    for (const op of plan) {
      try {
        executeOp(op);
        executed.push({ ...op, status: 'success' });
        updateManifest(op);
      } catch (e) {
        executed.push({ ...op, status: 'failed', error: e.message });
      
        // 报告状态
        reportPartialInstall(executed, plan);
      
        // 不自动回滚（可能回滚也会失败）
        // 而是记录状态，让用户决定
        saveRecoveryPlan(executed);
      
        throw new AiforgeError(
          `安装在第 ${executed.length}/${plan.length} 步失败\n` +
          `已安装: ${executed.filter(e => e.status === 'success').length} 项\n` +
          `失败: ${op.to} (${e.message})\n\n` +
          `运行 npx aiforge status 查看详情\n` +
          `运行 npx aiforge rollback 撤销已安装的文件`
        );
      }
    }
  }
```

**MVP 简化版：**

```
MVP 可能不需要完整的事务语义，但必须有:

  1. 预检查（Pre-flight check）:
     在任何文件操作前，验证所有目标路径可写
   
  2. 操作日志:
     实时输出每个文件的安装结果
       ✅ ~/.copilot/agents/api-dev.md
       ✅ ~/.copilot/agents/pm.md
       ❌ ~/.copilot/skills/tapd/ — 权限不足
       ⏸️ ~/.copilot/skills/git/ — 已跳过（前序失败）
       ⏸️ ~/.copilot/instructions/... — 已跳过

  3. 失败后不继续安装后续文件（fail-fast）:
     大部分失败原因（磁盘满、权限不足）不是偶发的
     继续尝试后续文件大概率也会失败

> ✅ 已采纳：architect.md §8.2 错误处理原则已调整为 fail-fast 策略。

  4. 失败时输出已安装文件列表:
     "以下文件已安装，如需撤销请手动删除:"
```

## 💥 攻击场景 2.4：跨平台路径地雷

```
地雷 1: 路径分隔符
──────────────────
  Windows: C:\Users\dev\.copilot\agents\
  Unix:    /home/dev/.copilot/agents/

  如果代码中硬编码了 '/':
    const target = homeDir + '/.copilot/agents/' + filename;
    → Windows 上生成: C:\Users\dev/.copilot/agents/api-dev.md
    → 混合分隔符，某些 Windows API 能处理，某些不能

  修复:
    始终使用 path.join():
    const target = path.join(homeDir, '.copilot', 'agents', filename);


地雷 2: home 目录
────────────────
  Linux:   ~  = /home/username
  macOS:   ~  = /Users/username
  Windows: ~  ≠ 一个标准位置
           USERPROFILE = C:\Users\username
           HOMEDIR     = C:\Users\username （有时）
           HOME        = 可能被 Git Bash 覆盖为 /c/Users/username

  os.homedir() 在大多数情况下正确，但:
    → 在 WSL 中: /home/username（不是 Windows 路径）
    → 在 Git Bash 中: 可能返回 POSIX 风格路径
    → 如果 USERPROFILE 被修改过（企业环境）: 可能指向网络路径

  修复:
    const home = os.homedir();
    if (!home || !fs.existsSync(home)) {
      throw new AiforgeError(
        '无法确定 home 目录\n' +
        `  os.homedir() 返回: ${home}\n` +
        '  请设置 AIFORGE_HOME 环境变量指定安装目录'
      );
    }


地雷 3: 文件名大小写
──────────────────
  macOS (APFS 默认): 大小写不敏感
    agents/API-Dev.md 和 agents/api-dev.md 是同一个文件

  Linux (ext4): 大小写敏感
    agents/API-Dev.md 和 agents/api-dev.md 是两个不同的文件

  如果仓库是在 Linux 上创建的，包含:
    agents/API-Dev.md
    agents/api-dev.md    ← 两个不同文件

  在 macOS 上克隆:
    只保留一个（git 会警告）
    →  aiforge 安装时只有一个文件
    → 但预期有两个

  修复:
    MVP 阶段: 不处理，这是 Git 自身的问题
    文档中注明: "建议知识仓库使用小写文件名，避免跨平台问题"


地雷 4: 长路径（Windows）
────────────────────
  Windows 传统 MAX_PATH = 260 字符
  典型场景:
    C:\Users\long-username\AppData\Local\Copilot\agents\very-long-skill-name\subsection\detailed-rules.md
    = 可能超过 260 字符

  结果: fs.mkdirSync() 静默失败或抛出难以理解的错误

  修复:
    Windows 上使用 \\?\ 前缀支持长路径:
    const safePath = process.platform === 'win32' 
      ? '\\\\?\\' + path.resolve(targetPath)
      : targetPath;
  
    或: 更好的方案是保持路径简短（这是 AI 工具的安装路径，一般不会太长）
    MVP 阶段: 不处理，但在错误处理中识别这个错误
```

## 💥 攻击场景 2.5：flatten 模式的内容转换陷阱

```
Cursor 的 flatten 场景:
  源: skills/tapd/
      ├── SKILL.md         (主文件)
      ├── examples.md      (示例)
      └── templates.md     (模板)

  目标: .cursor/rules/tapd.mdc  (单个文件)

  问题 1: 怎么从多个文件生成一个文件?
    方案 A: 只复制 mainFile（SKILL.md）,忽略其他文件
    方案 B: 把所有文件拼接成一个
    方案 C: 只复制 mainFile，但在文件中用注释标注"还有其他参考文件"

    如果选 A: 用户期望的 examples.md 和 templates.md 丢失了
    如果选 B: 拼接顺序是什么? 拼接后格式可能错乱
    如果选 C: AI 工具看到"其他参考文件"的注释但无法访问文件

  问题 2: .mdc 是什么格式?
    Cursor 的 .mdc 文件有特殊的 frontmatter:
    ---
    description: TAPD 工作流规范
    globs: 
    alwaysApply: true
    ---
    # 正文内容...

    如果 SKILL.md 没有这个 frontmatter:
      → Cursor 可能无法正确解析
      → 需要 aiforge 自动添加 frontmatter

    但 aiforge 怎么知道 description 应该写什么?
      → 从 SKILL.md 的标题推断?
      → 从目录名推断?
      → 留空让用户手动填?

  问题 3: 文件编码问题
    如果 SKILL.md 是 UTF-8 BOM 编码（Windows 常见）
    拼接后的 .mdc 文件中间会有不可见的 BOM 字符
    → 可能导致 Cursor 解析异常

暴露的脆弱点:
  flatten 模式涉及"内容转换"，这比单纯的复制/链接复杂得多
  每个 AI 工具的目标格式可能不同
  格式转换逻辑很容易出错
```

### 加固方案

```
MVP 策略: 极简 flatten

  1. 只复制 mainFile (SKILL.md → tapd.mdc)
  2. 不做内容转换（不加 frontmatter，不拼接）
  3. 文件扩展名按规则改为 .mdc
  4. 如果原文件已有 .mdc 格式的 frontmatter → 保持不变
  5. 如果原文件没有 frontmatter → 原样复制，不添加

  理由:
    知识仓库的维护者应该按目标工具的要求编写文件
    SKILL.md 如果是给 Cursor 用的，就应该自带正确的 frontmatter
    aiforge 不应该"猜"内容应该是什么格式

  这意味着: 知识仓库需要为不同工具准备不同格式的文件?
    → 不是。SKILL.md 可以用通用 markdown 格式。
    → 如果 Cursor 需要 frontmatter，仓库维护者在 SKILL.md 里加上。
    → 其他工具（Copilot）会忽略 yaml frontmatter，不影响。
```

## 这一轮的核心发现

```
发现 1 — manifest.json 是关键基础设施

  它不只是"记录装了什么"，而是整个系统的状态管理核心:
    - 冲突检测: 目标文件是 aiforge 的还是用户的?
    - 增量更新: 上次安装了什么? 这次有什么变化?
    - 健康检查: 符号链接是否还有效?
    - 回滚/清理: 需要撤销时删哪些文件?
    - 版本追踪: 上次安装的是哪个 commit?

  manifest.json 的优先级是 MVP 必须（不是 P1，而是 MVP 阻塞项）。


发现 2 — 符号链接模式比预想的复杂得多

  七种失败模式，每种都需要不同的处理策略。
  MVP 阶段的关键决策:
    ✅ 全局安装 (-g) 支持 -l 模式
    ❌ 项目级安装 (-p) 不支持 -l 模式
    ✅ Windows 上自动降级到复制模式
    ✅ 克隆路径固定到 ~/.aiforge/repos/


发现 3 — "不转换内容"是 MVP 的安全选择

  flatten 模式只做路径转换（改名 + 移动），不做内容转换。
  内容格式由知识仓库维护者负责。
  这避免了一整类内容转换相关的 bug。
```
