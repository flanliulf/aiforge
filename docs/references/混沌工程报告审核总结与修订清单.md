## 7. 统一审核模板

后续每篇混沌工程报告统一采用以下结构：

### 7.1 文档头信息

```
文档标题：混沌工程报告 #N：<组件名>
归档分组：A 组（MVP 风险）/ B 组（重构建议）/ C 组（未来预研）
基线文档：PRD.md / architect.md
审核状态：草稿 / 待评审 / 已确认
```

### 7.2 攻击场景结构

每个攻击场景包含：
- 攻击前提和触发条件
- 攻击过程和实际后果
- 可观测信号
- 加固方案（具体到类/方法/伪代码级别）
- 与已采纳 ADR 的一致性标注（`✅ 已采纳` 或 `⚠️ 注：当前已采纳 ADR-0XX`）

### 7.3 必须包含的收尾章节

1. 总结矩阵（攻击面 × 风险等级 × 防护措施）
2. 实现纳入判定表（攻击场景 × 纳入阶段 × 说明）
3. 与 PRD/ADD 的一致性检查

### 7.4 成文规范

- 不把"未来架构设想"写成"当前已定设计"
- 不跳过"与 PRD/ADD 是否冲突"的说明
- 所有与已采纳 ADR 冲突的建议必须添加显式标注
- 所有高优先级建议必须明确是：当前结论 / 重构建议 / 未来预研
