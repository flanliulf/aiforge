#### 问题 1：向量 #5 场景 5.4 和 5.5 的加固方案内容缺失

**文件：** `混沌工程攻击向量 #5：规则包生态系统.md`

**位置：** 场景 5.4（包发现和推荐的操纵）和场景 5.5（生态系统的长期健康）的加固方案部分

**现状：** 两个场景的加固方案仅列出了类名和方法名占位符，方法列表中出现大量 `if()` 无意义条目，实质内容完全缺失：

```
场景 5.4 加固方案（现状）：
  "主要方法（共 86 个）：
   - constructor()
   - calculateScore()
   - if()
   - if()
   - ... 及其他 76 个方法"

场景 5.5 加固方案（现状）：
  "主要方法（共 71 个）：
   - constructor()
   - if()
   - if()
   - ... 及其他 61 个方法"
```

**对比参考：** 同文件中场景 5.1、5.2、5.3 均有完整的伪代码和逻辑说明。

**需要补充的内容：**
- 场景 5.4：PackageReputationSystem 和 PackageSearchEngine 的核心逻辑，包括评分计算、异常检测、搜索排序的具体实现思路
- 场景 5.5：PackageLifecycleManager、PackageNamespace、DependencyConfusionGuard 的核心逻辑，包括健康检查、废弃处理、所有权转移的具体实现思路

#### 问题 2：向量 #5 场景 5.6 的加固方案错误复制了场景 5.1 的内容

**文件：** `混沌工程攻击向量 #5：规则包生态系统.md`

**位置：** 场景 5.6（规则包的互操作性和标准化）的加固方案部分

**现状：** 场景 5.6 的加固方案与场景 5.1（依赖解析）的加固方案**完全相同**，包括相同的伪代码：

```
function resolvePackage(name, constraint, depth):
    if depth > 100: throw "深度超限"
    if name in resolving: throw "循环依赖"
    ...
```

**问题：** 场景 5.6 讨论的是规则冲突、执行顺序、格式不兼容等互操作性问题，与依赖解析毫无关联，这是明显的内容错误复制。

**需要替换为：** 针对以下场景的加固方案：
- 规则包格式不兼容（manifest v1 vs v2）的兼容处理
- 规则冲突检测与优先级解决
- 规则执行顺序的拓扑排序
- 规则包与项目配置冲突的处理策略

#### 问题 3：`manifest.json` 与 `config.json` 命名混用，职责不清

**涉及文件：**
- `混沌工程攻击向量 #2：文件安装机制.md`
- `混沌工程攻击向量 #3：认证系统.md`

**现状：**
- 向量 #2 中，aiforge 的安装状态记录文件被称为 `~/.aiforge/manifest.json`
- 向量 #3 中，Token 配置文件被称为 `~/.aiforge/config.json`

两者都在 `~/.aiforge/` 目录下，但职责不同，文档中没有说明两者的关系。

**需要明确：**
- 这两个文件是否是同一个文件的不同字段？
- 还是两个独立文件，各自负责不同职责？
- 建议在某一向量中补充一段 `~/.aiforge/` 目录结构说明

---

#### 问题 4：`preflight`（预检查）概念在不同向量中定义不一致

**涉及文件：**
- `混沌工程攻击向量 #3：认证系统.md`（场景 3.4 的 preflight）
- `混沌工程攻击向量 #6：更新机制的安全边界.md`（更新前的验证流程）

**现状：**
- 向量 #3 的 `preflight` = Token 健康检查 + 权限验证 + 仓库访问验证
- 向量 #6 的更新前检查 = 签名验证 + 降级检查 + 链完整性验证

两者都是"执行前的安全检查"，但内容完全不同，且没有说明是否应该统一为一个 `preflight` 框架。

**需要明确：**
- 是否存在一个统一的 `preflight` 编排器，按场景调用不同的检查模块？
- 还是两个完全独立的检查系统，分别在不同入口触发？

**涉及文件：**
