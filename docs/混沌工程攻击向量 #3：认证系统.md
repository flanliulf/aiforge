## 💥 攻击场景 3.1：Token 存储与传输安全

```
Token泄露的五个渠道（按危险程度排序）:

渠道 1 — 进程列表（最危险）
─────────────────────────当前做法（危险）:
    git clone https://oauth2:${token}@gitlab.company.com/...
  任何用户执行 ps aux | grep git 都能看到完整 Token
  在共享服务器/CI环境中，其他用户可以看到攻击窗口: 从进程启动到结束（几秒到几分钟）


渠道 2 — Shell 历史（持久化泄露）
────────────────────────────────
  $ npx aiforge ... --token glpat-xxxx

  记录到:~/.bash_history
    ~/.zsh_history
    ~/.local/share/fish/fish_history

  攻击窗口: 永久（直到用户手动清除）


渠道 3 — Git 配置文件（持久化泄露）
──────────────────────────────────
  git clone后，.git/config 记录:
    [remote "origin"]
      url = https://oauth2:glpat-xxxx@gitlab.company.com/...

  攻击窗口: 永久（仓库存在期间）


渠道 4 — 错误日志（可能被分享）
──────────────────────────────
  git clone 失败时:
    fatal: Authentication failed for 'https://oauth2:glpat-xxxx@gitlab.company.com/...'

  用户可能:- 截图发到Slack 求助
    - 复制到 GitHub Issue
    - 记录到日志文件

  攻击窗口: 取决于日志保留时间和分享范围


渠道 5 — 明文配置文件（本地泄露）
────────────────────────────────
  ~/.aiforge/config.json:
    { "gitlab": { "token": "glpat-xxxx" } }

  如果文件权限是 644（默认）:
    → 同机器其他用户可读
  如果误提交到 Git:
    → 公开泄露

  攻击窗口: 永久
```

### 加固方案（分层防御）

**加固逻辑说明（共 6 层）：**

**层 1 — GIT_ASKPASS 机制（避免 Token 出现在命令行）**

创建临时 shell 脚本（内容为输出 Token），设置权限 700（仅 owner 可执行）；通过环境变量 GIT_ASKPASS 指向该脚本，让 git 调用它获取密码；从 URL 中移除已有认证信息，使用干净 URL 执行 git clone；设置 GIT_TERMINAL_PROMPT=0 禁止 git 弹出密码框；捕获 stdio 避免 Token 出现在终端；finally 块中立即删除临时脚本。

**层 2 — 生产模式下禁用 CLI Token 传递**

解析命令行参数时检测 --token 标志；若当前处于生产模式（`AIFORGE_MODE=production` 或 `config.json` 中 `"mode": "production"`），则输出安全警告（说明 Token 会出现在命令行历史和进程列表中），提示使用环境变量或配置命令替代，然后退出进程。开发/调试模式下允许使用 --token，但输出安全提示。

**层 3 — 系统密钥链存储（keytar）**

- `saveToken`：调用 keytar.setPassword 存储 Token；配置文件只记录元数据（useKeychain: true、configuredAt）
- `getToken`：优先从密钥链读取 → 降级到环境变量 GITLAB_TOKEN（附带警告，建议使用配置命令）→ 最后交互式输入

**层 4 — 配置文件权限检查**

创建配置目录时设置权限 700；检查配置文件权限，若不是 600 则输出警告并自动修复（chmod 600）。

**层 5 — 错误信息脱敏**

定义正则模式匹配各种 Token 格式（GitLab glpat-、GitHub ghp_/gho_/ghu_、URL 中的认证信息、Header 中的 PRIVATE-TOKEN）；对每个匹配项保留前缀，将 Token 值替换为 ****；注册全局 uncaughtException 处理器，确保所有错误输出都经过脱敏。

**层 6 — 交互式输入（不回显）**

设置终端为 raw mode 逐字符读取；不显示输入字符（不回显）；支持退格键删除、Ctrl+C 退出；回车键确认后恢复终端模式并返回 Token。

## 💥 攻击场景 3.2：Token 权限问题

```
GitLab Token 的权限范围（scopes）:- api: 完全API 访问（危险）
  - read_api: 只读 API
  - read_repository: 读取仓库
  - write_repository: 写入仓库（不需要）
  - read_user: 读取用户信息
  - sudo: 管理员权限（极度危险）

aiforge 需要的最小权限:✓ read_api: 验证 Token（调用 /api/v4/user）
  ✓ read_repository: 克隆仓库


场景 A — 权限不足（功能失败）
──────────────────────────────
  用户创建 Token 时只勾选了 read_user
  行为:
    /api/v4/user → 200 OK（read_user 足够）
    git clone → 403 Forbidden（缺少 read_repository）

  问题:
    Token 验证通过，但克隆失败
    错误信息来自 git: "fatal: Authentication failed"用户不知道是权限问题


场景 B — 仓库级权限不足
──────────────────────
  Token 有 read_repository 权限
  但用户不是目标仓库的成员

  行为:
    /api/v4/user → 200 OK
    /api/v4/projects/team%2Fai-base → 404 Not Found
    （GitLab 对无权限的私有仓库返回 404，不是 403）

  问题:
    用户以为"仓库不存在"
    实际是"仓库存在但你没权限"


场景 C — 权限过大（安全风险）
──────────────────────────────
  用户使用了 admin Token（有 api + sudo 权限）

  风险:
    aiforge 只需要 read_repository
    但拿到了管理员级别的 Token
    如果 Token 泄露:
      →攻击者可以修改/删除任何仓库
      → 攻击者可以创建/删除用户
      → 攻击者可以修改系统配置

    影响范围: 整个 GitLab 实例


场景 D — 跨项目复用高权限 Token
──────────────────────────────
  用户有一个"万能 Token"，用于所有工具
  这个 Token 有多个项目的 write权限

  风险:
    aiforge 只需要访问知识仓库（只读）
    但 Token 能写入公司所有项目

    如果 aiforge 有漏洞或被恶意修改:
      → 可以修改用户有权限的所有仓库
      → 数据泄露/破坏范围扩大
```

### 加固方案

**加固逻辑说明：**

**`validateTokenPermissions`（Token 权限验证）**

```
调用 GET /api/v4/personal_access_tokens/self（携带 PRIVATE-TOKEN 头）
若 401 → 抛出"Token 无效或已过期"错误

检查必需权限 [read_api, read_repository]：
  若有缺失 → 抛出详细错误（列出缺少的权限 + 修复步骤：访问 Token 设置页面重新创建）

检查多余权限（不在必需列表中的）：
  若存在 → 输出安全建议警告
  若包含危险权限 [api, write_repository, sudo] → 输出高危警告（说明泄露影响范围）
```

**`validateRepoAccess`（仓库访问验证）**

```
调用 GET /api/v4/projects/{encoded_repoPath}（携带 PRIVATE-TOKEN 头）
若 404 → 抛出错误（列出可能原因：路径错误/私有仓库/已删除 + 修复步骤）
若 403 → 抛出错误（Token 有权限但不是仓库成员，建议联系管理员添加）
若 200 → 返回项目信息
```

## 💥 攻击场景 3.3：仓库访问判断

```
场景整合了"公开 vs 私有判断"和"仓库级权限"问题


场景 A — 误判为"仓库不存在"
──────────────────────────
  URL: https://gitlab.company.com/team/ai-base.git

  无认证尝试:
    git clone → 404 Not Found

  问题:
    GitLab 对私有仓库返回 404（而不是 401）
    → 无法区分"不存在"和"需要认证"
    → 用户: "仓库明明在啊!"


场景 B — 对公开仓库误用高权限 Token
──────────────────────────────────
  URL: https://github.com/public-org/open-ai-base.git
  用户配置了 GITHUB_TOKEN（有 write权限）

  aiforge 检测到 Token → 使用 Token 克隆

  风险:
    公开仓库不需要任何权限
    但用了一个高权限 Token
    如果 Token 泄露 → 影响范围扩大


场景 C — GitLab "internal" 可见性
─────────────────────────────────
  GitLab 有三种可见性:- public: 所有人可见
    - internal: 登录用户可见
    - private: 成员可见

  对internal 仓库:
    无认证:404
    有认证但不是成员: 200（能看到但可能不能 clone）

  判断逻辑变得复杂
```

### 加固方案

**加固逻辑说明：**

**`authenticateAndClone`（智能认证克隆流程）**

```
步骤 1：尝试无认证克隆（检测是否公开仓库）
  若成功 → 直接返回（公开仓库，无需 Token）

步骤 2：获取 Token（调用 getToken）
  若无 Token → 抛出详细错误（说明可能原因 + 如何配置 Token）

步骤 3：执行预检查（preflight）

步骤 4：使用 Token 克隆（cloneWithToken），返回克隆路径
```

**`tryCloneWithoutAuth`（无认证快速测试）**

创建临时目录，执行 `git clone --depth 1`（10 秒超时，捕获 stdio）；成功则删除临时目录返回 `{success: true}`；失败则返回 `{success: false}`。

## 💥 攻击场景 3.4：Token 生命周期管理

```
场景 A — Token 已过期
─────────────────────
  GitLab Token 默认过期时间: 30/60/90 天（取决于实例配置）

  用户三个月前设置了 Token
  现在运行 aiforge:
    /api/v4/user → 401 Unauthorized

  问题:
    用户不知道是"过期"还是"Token 错了"
    错误信息不够明确


场景 B — Token 被撤销
─────────────────────
  用户或管理员在 GitLab 上撤销了 Token
  但 ~/.aiforge/config.json 中还保存着

  行为: 同场景 A

  问题:
    aiforge 不知道 Token 已被撤销
    继续使用无效 Token


场景 C — Token 即将过期（7 天内）
────────────────────────────────
  Token 还能用，但快过期了
  用户不知道，继续使用
  某天突然失败→ "昨天还好好的"

  问题:
    没有过期预警机制
    用户被动等到失败才知道


场景 D — Token 权限被修改
────────────────────────
  Token 创建时有 read_repository 权限
  管理员后来修改了权限范围（GitLab 企业版支持）

  行为:
    Token 验证通过（read_api还在）
    但克隆失败（read_repository 被移除）

  问题:
    用户不知道权限被修改了


场景 E — Token 永不过期的风险
───────────────────────────
  用户创建 Token 时选择"永不过期"

  风险:
    Token泄露后，攻击者可以长期使用
    没有自动失效机制
    用户可能忘记这个 Token 的存在
```

### 加固方案

**加固逻辑说明：**

**`checkTokenHealth`（Token 健康检查）**

```
调用 GET /api/v4/personal_access_tokens/self（携带 PRIVATE-TOKEN 头）
若 401 → 返回 {valid: false, reason: "Token 无效或已过期"}

若有 expires_at：
  计算剩余天数 = (expires_at - now) / 一天毫秒数
  若 < 0  → 返回 {valid: false, reason: "Token 已于 {date} 过期"}
  若 ≤ 7  → 返回 {valid: true, warning: "Token 将在 N 天后过期", action: "建议尽快更新"}
若无 expires_at（永不过期）：
  返回 {valid: true, warning: "Token 设置为永不过期", action: "建议设置过期时间"}

API 调用失败 → 返回 {valid: true, warning: "无法验证 Token 状态"}（不阻塞主流程）
```

**`preflight`（完整预检查编排）**

依次执行：
1. Token 健康检查 → 无效则抛出含修复步骤的错误；有警告则输出
2. Token 权限验证（调用 validateTokenPermissions）
3. 仓库访问验证（调用 validateRepoAccess），输出确认信息

> **术语说明：** 本文中的 `preflight` 特指认证系统的预检查——验证 Token 健康状态和权限。与攻击向量 #6 中的更新 preflight（验证签名和完整性）是两个独立的检查系统，各自在不同入口触发：认证 preflight 在 `resolveAuth()` 流程中执行，更新 preflight 在 `aiforge update` 命令中执行。

## 💥 攻击场景 3.5：多 GitLab 实例管理

```
企业环境中的典型场景:
  - 公司内部: gitlab.company.com
  - 客户实例: gitlab.client-a.com, gitlab.client-b.com
  - 个人实例: gitlab.com


场景 A — 环境变量只能设置一个
──────────────────────────────
  export GITLAB_TOKEN=xxx

  问题:
    只能用于一个实例
    访问其他实例时需要手动切换
    容易忘记切换 → 用错 Token


场景 B — Token 混淆
──────────────────
  用户有多个 Token，不记得哪个是哪个

  ~/.aiforge/config.json:
    { "gitlab": { "token": "glpat-xxxx" } }
    ← 这是哪个实例的 Token?
  用公司的 Token 访问客户的 GitLab:
    → 401 Unauthorized
    → 用户: "我明明设置了 Token"


场景 C — Token 管理困难
──────────────────────
  用户想知道:
    - 我配置了哪些实例的 Token?
    - 某个 Token 是什么时候配置的?
    - 某个 Token 对应哪个 GitLab 用户?

  当前没有管理工具


场景 D — Token 轮换困难
──────────────────────
  公司要求每 90 天轮换 Token
  用户有 5 个 GitLab 实例

  问题:
    不知道哪些 Token 快过期了
    需要逐个登录 GitLab 检查
    容易遗漏
```

### 加固方案

**加固逻辑说明：**

**配置文件结构**

按 hostname 为键存储各实例信息，每个实例记录：`useKeychain`（是否使用密钥链）、`user`（用户名）、`configuredAt`（配置时间）。示例：

```json
{
  "gitlab": {
    "gitlab.company.com": { "useKeychain": true, "user": "zhangsan", "configuredAt": "2024-01-20T15:30:22Z" },
    "gitlab.client-a.com": { "useKeychain": true, "user": "zhangsan", "configuredAt": "2024-01-15T10:20:15Z" }
  }
}
```

**`getTokenForRepo`（Token 自动路由）**

从仓库 URL 提取 hostname → 从密钥链按 hostname 精确匹配读取 Token → 降级到环境变量 GITLAB_TOKEN（附带警告，建议配置专用 Token）→ 交互式输入 → 询问是否保存到密钥链。

**Token 管理命令逻辑：**

- `list-tokens`：遍历配置文件中所有实例，对每个实例从密钥链读取 Token 并调用 checkTokenHealth，输出用户名、配置时间、存储方式、有效状态（含警告）
- `set-token <host>`：提示输入 Token → 调用 checkTokenHealth 验证（失败则抛错）→ 验证通过后调用 saveToken 保存 → 输出 Token 信息（用户名、权限、过期时间）
- `remove-token <host>`：若未指定 host 则交互式列表选择；确认后从密钥链删除 Token，并从配置文件删除对应条目
- `test-token <host>`：从密钥链读取 Token → 调用 checkTokenHealth → 输出有效性、用户名、权限、过期时间及剩余天数

## 💥 攻击场景 3.6：网络环境问题

```
场景 A — 企业代理
────────────────
  公司网络要求所有 HTTP 请求通过代理
  环境变量: HTTP_PROXY=http://proxy.company.com:8080

  问题 1: git clone 可能不遵守 HTTP_PROXY→ 需要配置 git config http.proxy

  问题 2: 代理需要认证
    HTTP_PROXY=http://user:pass@proxy.company.com:8080
    → 密码可能包含特殊字符，需要 URL 编码

  问题 3: HTTPS 代理
    某些代理只支持 HTTP，不支持 HTTPS
    → git clone https://... 失败


场景 B — VPN 要求
────────────────
  GitLab 实例只能通过 VPN 访问
  用户忘记连 VPN 就运行 aiforge

  行为:
    fetch(gitlab.company.com/api/v4/user)
    → ENOTFOUND 或 ETIMEDOUT

  错误信息:
    "getaddrinfo ENOTFOUND gitlab.company.com"
    → 用户不知道是 VPN 问题还是 URL 错误


场景 C — 自签名证书
────────────────
  企业内部 GitLab 使用自签名 SSL 证书

  行为:
    fetch(https://gitlab.company.com/...)
    → Error: self signed certificate in certificate chain

  问题:
    Node.js 默认拒绝自签名证书
    用户需要:
      - 设置 NODE_TLS_REJECT_UNAUTHORIZED=0（不安全）
      - 或导入证书到系统信任链（复杂）


场景 D — 防火墙限制
──────────────────
  公司防火墙只允许特定端口
  GitLab 使用非标准端口（如 8443）

  行为:
    连接超时

  错误信息不明确


场景 E — DNS 解析问题
────────────────────
  内部 GitLab 使用内网 DNS
  用户在外网环境运行 aiforge

  行为:
    ENOTFOUND

  用户不知道是 DNS 问题
```

### 加固方案

**加固逻辑说明：**

**代理支持**

- `createFetchOptions`：检测环境变量 HTTPS_PROXY 或 HTTP_PROXY，若存在则创建 HttpsProxyAgent 并设置到 fetch options；日志输出时对代理 URL 中的认证信息脱敏（替换 `//user:pass@` 为 `//***@`）
- `fetchWithProxy`：合并代理 options 后调用 fetch
- `configureGitProxy`：若检测到代理环境变量，执行 `git config --global http.proxy` 配置 Git 代理（失败则输出警告，不阻塞）

**`handleNetworkError`（网络错误友好提示）**

```
若 ENOTFOUND  → 提示"无法解析域名"，列出可能原因（URL 错误/需要 VPN/DNS 问题）+ 修复步骤
若 ETIMEDOUT  → 提示"连接超时"，列出可能原因（网络不稳定/服务无响应/防火墙/需要代理）+ 修复步骤
若 ECONNREFUSED → 提示"连接被拒绝"，列出可能原因（服务未运行/端口错误/防火墙）+ 修复步骤
若 self signed certificate → 提示"SSL 证书验证失败"，提供三种解决方案（安装受信任证书/导入系统信任链/临时禁用验证）
若 certificate has expired → 提示"SSL 证书已过期"，建议联系管理员更新
其他 → 通用网络错误提示（经过脱敏处理）
```

**`checkNetworkConnectivity`**

发送 HEAD 请求（5 秒超时）测试连接；失败则调用 handleNetworkError 转换为友好错误。

## 💥 攻击场景 3.7：Token 泄露后的影响面

```
假设: Token 通过任何渠道泄露了


影响范围取决于 Token 权限:

情况 A — api 权限（完全访问）
────────────────────────攻击者可以:✓ 读取所有用户能访问的仓库
    ✓ 修改/删除仓库
    ✓ 创建/删除用户（如果是 admin）
    ✓ 修改用户设置
    ✓ 创建/删除 SSH 密钥
    ✓ 冒充用户进行任何操作

  影响面: 极大（整个 GitLab 实例）


情况 B — read_api + read_repository
──────────────────────────────────
  攻击者可以:
    ✓ 读取所有用户能访问的仓库
    ✗ 不能修改任何内容

  影响面: 中等（数据泄露，但不能破坏）


情况 C — write_repository
────────────────────────
  攻击者可以:
    ✓ 读取仓库
    ✓ 提交恶意代码
    ✓ 修改/删除分支
    ✓ 修改 CI/CD 配置

  影响面: 高（代码投毒、供应链攻击）


影响持续时间取决于过期设置:有过期时间（30 天）:
    → 影响面有限（30 天后自动失效）
    → 但 30 天内攻击者可以持续访问

  永不过期:
    → 影响面永久（除非用户手动撤销）
    → 用户可能不知道 Token 泄露，永远不撤销


真实攻击场景:

  场景 1 — 代码仓库投毒
    攻击者获得 write_repository Token
    → 在热门开源项目中注入恶意代码
    → 影响所有使用该项目的下游用户
    → 供应链攻击

  场景 2 — 数据窃取
    攻击者获得 read_repository Token
    → 克隆所有私有仓库
    → 窃取商业机密、客户数据、API 密钥
    → 出售或勒索

  场景 3 — CI/CD 劫持
    攻击者获得 write_repository Token
    → 修改 .gitlab-ci.yml
    → 在 CI 中执行恶意代码
    → 窃取 CI 环境中的密钥（AWS、数据库等）

  场景 4 — 横向移动
    攻击者获得一个项目的 Token
    → 发现 Token 能访问其他项目（权限过大）
    → 横向移动到更敏感的项目
    → 扩大攻击面
```

### 防御策略（整合到文档）

**Token 安全最佳实践**

**创建 Token 时**

✅ 最小权限原则：aiforge 只需要 read_api + read_repository，不要授予 api、write_*、sudo

✅ 设置过期时间：推荐 30-90 天，不要选择永不过期

✅ 使用描述性名称：如 "aiforge-macbook-pro-2024"（方便识别和撤销），避免使用 "token1" 等无意义名称

✅ 为不同用途创建不同 Token：aiforge 专用、CI/CD 专用，不要一个 Token 用于所有工具

**使用 Token 时**

✅ 使用系统密钥链存储：`npx aiforge config set-token`

✅ 或使用环境变量（临时）：`export GITLAB_TOKEN=xxx`（不要写入 .bashrc 持久化）

❌ 不要：在命令行参数中传递 Token、在代码中硬编码 Token、在配置文件中明文存储 Token、把 Token 提交到 Git 仓库

**Token 泄露时的应急响应**

1. 立即撤销 Token：GitLab → User Settings → Access Tokens → 找到可疑 Token → Revoke
2. 检查 Token 使用记录：GitLab → Admin → Audit Events，查看是否有异常访问
3. 创建新 Token：使用更严格的权限，设置更短的过期时间
4. 更新所有使用该 Token 的系统：`npx aiforge config set-token`，更新 CI/CD 变量

**定期维护**

✅ 每 90 天轮换 Token（即使没有泄露，也定期更换，降低长期泄露风险）

✅ 定期审计 Token：`npx aiforge config list-tokens`，删除不再使用的 Token

✅ 监控 Token 使用：检查 GitLab Audit Events，发现异常立即撤销

## 核心代码架构
**核心模块架构说明：**

**TokenManager 类**

- `getToken(hostname)`：按优先级获取 Token：① 密钥链（keytar.getPassword）→ ② 环境变量 GITLAB_TOKEN（附带警告）→ ③ 交互式输入（promptForToken）
- `saveToken(hostname, token)`：调用 keytar.setPassword 存储；更新配置文件元数据（useKeychain: true、configuredAt）
- `promptForToken(hostname)`：交互式不回显输入

**PreflightChecker 类**

- `check(gitlabUrl, token, repoPath)`：编排完整预检查流程：① Token 健康检查（无效则抛错，有警告则输出）→ ② 权限验证 → ③ 仓库访问验证（输出确认信息）
- `checkTokenHealth(gitlabUrl, token)`：调用 /api/v4/personal_access_tokens/self；处理 401（无效）、过期（daysLeft < 0）、即将过期（daysLeft ≤ 7）、永不过期四种情况
- `validatePermissions(gitlabUrl, token, tokenInfo)`：检查必需权限 [read_api, read_repository]，缺失则抛错；检查多余权限，存在则警告
- `validateRepoAccess(gitlabUrl, token, repoPath)`：调用 /api/v4/projects/{id}；处理 404（不存在/无权限）、403（无成员资格）两种错误情况

**GitCloner 类**

- `clone(repoUrl, token, targetDir)`：创建临时 askpass 脚本（权限 700）→ 清理 URL 中的认证信息 → 通过 GIT_ASKPASS 环境变量执行 git clone（捕获 stdio）→ finally 块删除临时脚本；catch 块对错误信息脱敏后重新抛出

**Error Handler 模块**

- `sanitizeError(message)`：用正则替换消息中的各种 Token 格式（glpat-、ghp_、URL 认证信息、PRIVATE-TOKEN Header），保留前缀，值替换为 ****
- `handleNetworkError(error, gitlabUrl)`：按错误码（ENOTFOUND/ETIMEDOUT/ECONNREFUSED）和错误消息（self signed certificate/certificate has expired）分类，转换为友好的中文错误提示
- 注册全局 uncaughtException 处理器，对所有未捕获异常进行脱敏处理

**主流程 `installFromGitLab`**

```
1. 解析仓库 URL，提取 hostname 和 repoPath
2. TokenManager.getToken(hostname) 获取 Token
3. PreflightChecker.check(gitlabUrl, token, repoPath) 预检查
   （网络错误通过 handleNetworkError 转换为友好提示）
4. GitCloner.clone(repoUrl, token, targetDir) 克隆仓库
5. 继续安装流程，返回克隆路径
```

## 这一轮的核心发现（最终总结）

认证系统的攻击揭示了三个层次的问题：
技术层：Token 在传输和存储过程中的泄露风险
流程层：Token 的生命周期管理（创建、使用、过期、撤销）
人员层：用户对安全最佳实践的理解和遵守


```
发现 1 — Token泄露是"系统性风险"，不是"单点问题"
═══════════════════════════════════════════════════════
Token 可以通过 5 个独立渠道泄露:进程列表 → 命令行参数
    Shell 历史 → --token 参数
    Git 配置 → URL嵌入
    错误日志 → 异常信息
    配置文件 → 明文存储

  修复策略必须是"分层防御":✅ GIT_ASKPASS（避免命令行暴露）
    ✅ 生产模式禁用 --token（避免历史记录，开发模式允许）
    ✅ 错误脱敏（避免日志泄露）
    ✅ 密钥链存储（避免文件泄露）
    ✅ 文件权限 0o600（最后防线）

  任何一层失效，其他层仍能提供保护


发现 2 — "预检查"比"失败后报错"更重要
═══════════════════════════════════════

  传统流程（被动）:
    开始操作 → 失败 → 报错 → 用户修复 → 重试
    → 用户已经等了几分钟→ 错误信息来自底层工具（git），不友好

  预检查流程（主动）:
    预检查 → 发现问题 → 精确提示 → 用户修复 → 开始操作
    → 在几秒内发现问题
    → 错误信息由aiforge 控制，友好且可操作

  预检查内容:
    ✅ Token 有效性（过期/撤销）
    ✅ Token 权限（scopes）
    ✅ 仓库访问权限（404vs 403）
    ✅ 网络连接（VPN/代理/SSL）
  投入: 3-5 个API 调用（几秒）
  回报: 避免 90% 的失败和困惑


发现 3 —多实例支持是"必需"，不是"可选"
═══════════════════════════════════════════

  最初假设: "大部分用户只用一个 GitLab"
  实际情况: 企业用户通常有 3-5 个实例
    - 公司内部 GitLab
    - 客户/合作伙伴 GitLab
    - 个人 GitLab/GitHub

  如果不支持多实例:
    → 用户需要频繁切换环境变量
    → 容易用错 Token（公司 Token 访问客户 GitLab）
    → 体验极差

  MVP 必须实现:
    ✅ 按hostname 存储 Token
    ✅ 自动路由（根据 URL 选择 Token）
    ✅ 管理命令（list/set/remove/test）


发现 4 — 网络错误提示是"隐形的用户体验"
═══════════════════════════════════════════

  技术视角: 网络错误不是 aiforge 的问题
  用户视角: "aiforge 不好用，连不上"

  Node.js 原生错误:
    "Error: getaddrinfo ENOTFOUND gitlab.company.com"
    → 用户: "什么是 getaddrinfo？"

  友好错误:
    "无法解析域名: gitlab.company.com可能需要连接 VPN 或检查 DNS 配置"
    → 用户: "哦，我忘了连VPN"

  这种"翻译"工作:
    - 技术成本低（几个 if 判断）
    - 用户体验提升巨大
    - 减少 80% 的支持请求


发现 5 — 安全是"教育"问题，不只是"技术"问题
═══════════════════════════════════════════════════

  技术手段只能防止部分风险:
    ✅ GIT_ASKPASS → 防止进程列表泄露
    ✅ 密钥链 → 防止文件泄露
    ✅ 错误脱敏 → 防止日志泄露

  但很多风险来自用户行为:
    ❌ 使用权限过大的 Token（api, sudo）
    ❌ Token 永不过期
    ❌ 一个 Token 用于所有工具
    ❌ 在代码中硬编码 Token

  需要通过"教育"改变行为:
    ✅ 运行时警告（检测到不安全行为）
    ✅ 权限审计（Token 有不必要的权限）
    ✅ 文档中的最佳实践✅ 示例配置（展示正确做法）

  目标: 让用户理解"为什么"，而不只是"怎么做"


发现 6 — CI/CD 是独立使用场景，需要专门适配
═══════════════════════════════════════════════

  本地环境 vs CI/CD 环境:
    持久化存储 vs 临时容器
    单用户 vs 可能并发
    交互式 vs 非交互式
    长期使用 vs 一次性

  CI/CD 特殊需求:
    ✅ 检测 CI环境（CI=true 等）
    ✅ 警告不合理参数（-g 在 CI 中）
    ✅ 使用项目相对路径（避免污染 runner）
    ✅ 并发安全（文件锁）✅ CI 特定的 Token 配置（CI_JOB_TOKEN）

  如果不适配:
    → 全局安装在容器中无意义
    → 并发 job 可能冲突
    → Token 配置困难
```
