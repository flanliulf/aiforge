 🎯 混沌工程攻击向量 #1: 双层检测机制

  我们的设计:
  Layer 1 (智能检测): 自动扫描目录,匹配 DETECTION_PATTERNS
  Layer 2 (显式声明): aiforge.json 的 structure 映射

  让我们开始攻击:

  💥 攻击场景 1.1: 检测冲突

  如果一个目录同时匹配多个模式会怎样?

  仓库结构:
  ├── agents-and-skills/     # 既匹配 /agents?$/ 又匹配 /skills?$/
  │   ├── api-dev.agent.md
  │   └── git.skill.md
  └── aiforge.json (不存在)

  问题:
    DETECTION_PATTERNS 的"第一个匹配获胜"规则
    → agents-and-skills/ 被识别为 'agent'
    → git.skill.md 被错误地当作 agent 安装
    → 用户困惑: "为什么我的 skill 没装上?"

  暴露的脆弱点:
  - 目录独占性原则在"故意混合命名"时失效
  - 没有警告机制告诉用户"这个目录名有歧义"
  - 错误是"静默的"(文件被安装到错误位置,但没有报错)

  需要加固:
  - 检测到歧义目录名时,输出警告
  - 提供 --strict 模式,歧义时直接失败
  - 或者:扫描目录内的文件特征来"二次确认"类型

  加固方案:

  1. 歧义检测算法:
     对每个目录名运行所有 DETECTION_PATTERNS
     如果匹配到 2 个以上类型 → 标记为歧义目录

  2. 歧义处理策略:
     默认模式: 输出警告 + 使用第一个匹配 + 继续安装
       "⚠️ 目录 'agents-and-skills/' 同时匹配 agent 和 skill 类型"
       "  当前按 agent 类型处理，如需指定请在 aiforge.json 中声明"
     --strict 模式: 输出错误 + 终止安装
       "❌ 目录名歧义，请在 aiforge.json 中明确声明类型"

  3. 文件特征二次确认:
     匹配到 agent 类型后，扫描目录内文件:
       if 目录内有 *.agent.md → 确认为 agent
       if 目录内有 skill.md 子目录结构 → 确认为 skill
       if 两者都有 → 保持歧义警告
     这是一个轻量级的启发式检查，不替代 aiforge.json 的显式声明

  ---
  💥 攻击场景 1.2: 检测遗漏

  如果团队使用了完全不在 DETECTION_PATTERNS 中的目录名会怎样?

  仓库结构:
  ├── ai-prompts/           # 不匹配任何模式
  │   ├── api-dev.agent.md
  │   └── pm.agent.md
  └── aiforge.json (不存在)

  问题:
    Layer 1 检测失败 → 跳过这个目录
    没有 Layer 2 配置 → 整个仓库"看起来是空的"
    用户: "为什么什么都没装?"

  暴露的脆弱点:
  - 零配置路径在"非标准命名"时完全失效
  - 没有"发现了目录但不知道是什么类型"的提示
  - 用户不知道是"仓库真的是空的"还是"检测失败了"

  需要加固:
  - 输出"发现但未识别的目录"列表
  - 提示用户"如果这些目录包含配置,请在 aiforge.json 中声明"
  - 或者:提供 npx aiforge doctor 命令诊断仓库

  ---
  💥 攻击场景 1.3: Layer 2 覆盖的意外后果

  如果用户在 aiforge.json 中声明了错误的映射会怎样?

  {
    "structure": {
      "agent": {
        "dirs": ["skills"],        // 错误: 把 skills 目录映射为 agent
        "pattern": "*.md"
      }
    },
    "overrideDetection": true      // 完全禁用自动检测
  }

  问题:
  - Layer 1 被禁用
  - Layer 2 的错误配置导致所有 skills 被当作 agents 安装
  - 用户可能很久之后才发现问题(因为文件"确实被安装了",只是位置错了)

  暴露的脆弱点:
  - overrideDetection: true 是一个"核武器"开关,没有安全网
  - 没有验证 structure 映射的"合理性"
  - 没有"试运行"机制让用户预览检测结果

  需要加固:
  - 在 overrideDetection: true 时输出警告
  - 提供 npx aiforge detect 命令预览检测结果
  - 或者:检查文件内容是否与声明的类型匹配(如 *.agent.md 应该在 agent 目录)

  加固方案:

  1. overrideDetection 安全网:
     当 overrideDetection: true 时，强制输出检测结果对比:
       "⚠️ 自动检测已被 aiforge.json 覆盖"
       "  Layer 1 检测结果: agents/ → agent, skills/ → skill"
       "  aiforge.json 声明:  skills/ → agent (与自动检测不一致)"
       "  如果这是有意为之，请忽略此警告"

  2. structure 映射合理性校验:
     对 aiforge.json 中声明的每个映射，交叉验证文件扩展名:
       agent 类型目录 → 期望包含 *.agent.md 文件
       skill 类型目录 → 期望包含 skill.md 或子目录结构
       instruction 类型目录 → 期望包含 *.instructions.md 文件
     不匹配时输出警告（不阻塞安装）:
       "⚠️ structure.agent.dirs 中的 'skills/' 不包含 *.agent.md 文件"
       "  该目录包含: skill.md, README.md"
       "  请确认类型声明是否正确"

  3. 与 --dry-run 联动:
     npx aiforge --dry-run 时，额外输出完整的检测过程:
       Layer 1 结果 → Layer 2 覆盖 → 最终映射
     让用户在实际安装前预览检测结果

## 回答三个问题

### 问题 1：这些场景真实吗？

```
场景 1.1（检测冲突）: 不太真实
  "agents-and-skills/" 这种命名几乎不会出现在生产仓库中。
  没有团队会故意把两种资源混在一个目录里并用连字符拼接两个语义词。
  这是一个理论上的边界条件，实际概率极低。

场景 1.2（检测遗漏）: 非常真实 ⚠️
  "ai-prompts/" 这种命名完全可能出现。
  尤其是在 aiforge 还没有广泛采用时，
  团队的目录命名完全是自发的，不会去参考 DETECTION_PATTERNS。
  这是最高概率的真实场景。

场景 1.3（错误覆盖）: 真实但低概率
  写 aiforge.json 的人是仓库维护者（Tech Lead），
  他们有足够的技术能力，不太会把 skills 写成 agent。
  但"复制粘贴时犯错"是可能的。
  更现实的场景是：仓库重构后 aiforge.json 忘了更新。
```

### 问题 2：哪个最危险？

**场景 1.2（检测遗漏）最危险。** 原因不是技术后果严重，而是**用户体验后果最恶劣**：

```
场景 1.1 的后果:
  文件装到错误位置 → 用户打开 AI 工具时发现"不对" → 排查后能发现
  → 后果: 需要排查，但最终能定位

场景 1.2 的后果:
  什么都没装 → 用户以为"工具坏了"或"仓库是空的"
  → 不会往"目录名不匹配"这个方向想
  → 大概率的反应: "这个工具不好用，算了"
  → 后果: 用户流失。静默失败是最糟糕的失败模式。

场景 1.3 的后果:
  文件装到错误位置 → 类似 1.1 但能通过 diff 发现
  → 后果: 有 aiforge.json 意味着用户能力较强，自行修复概率高
```

**排序：1.2 > 1.1 > 1.3**

静默的"什么都没做"比"做错了"更危险，因为"做错了"至少产生了可观察的异常。

## 补充攻击场景

### 💥 攻击场景 1.4：大仓库的性能退化

```
仓库结构:
├── agents/          (5 个文件)
├── skills/          (20 个技能目录)
├── instructions/    (3 个文件)
├── docs/            (200 个文件)
├── examples/        (500 个文件)
├── archives/        (1000 个文件)
├── .git/            (对象数据库)
└── node_modules/    (不太可能，但如果有呢？)

DETECTION_PATTERNS 的扫描行为:
  1. 列出根目录所有子目录
  2. 对每个目录匹配所有模式
  3. 匹配成功后，扫描目录内所有文件

问题:
  根目录有 8 个子目录 → 扫描 8 次，可接受
  但如果仓库是 monorepo 结构:
  
  ├── packages/
  │   ├── service-a/
  │   │   └── ai-config/
  │   │       └── agents/       ← 嵌套在深层
  │   ├── service-b/
  │   │   └── ai-config/
  │   │       └── agents/
  │   └── ... (50 个 service)

  检测只扫描根目录一层？
    → 找不到嵌套的 agents/
    → 又是一个"静默失败"

  检测递归扫描所有层级？
    → 性能灾难
    → 还可能匹配到不相关的目录（比如 node_modules 里的 agents/）

暴露的脆弱点:
  DETECTION_PATTERNS 的扫描深度没有定义。
  扫描根目录一层太浅，递归扫描太深。
  没有排除规则（.git/, node_modules/, dist/）。
```

**需要加固：**

```
明确的设计决策:
  扫描深度 = 仅根目录一层（MVP 阶段）
  原因: 知识仓库的设计意图就是"顶层目录即分类"
  如果仓库的资源在深层 → 必须用 aiforge.json 显式声明

排除列表:
  const EXCLUDED_DIRS = [
    '.git', 'node_modules', 'dist', 'build', 
    '.vscode', '.idea', '__pycache__'
  ];

UX 处理:
  如果扫描后发现 0 个匹配:
    ⚠️ 未在根目录发现匹配的配置目录
  
    发现但未识别的目录:
      packages/    docs/    examples/
  
    如果你的配置在子目录中，请创建 aiforge.json:
      { "structure": { "agent": { "dirs": ["packages/shared/agents"] } } }
```

### 💥 攻击场景 1.5：aiforge.json 本身的问题

```
场景 A: JSON 语法错误
─────────────────────
{
  "structure": {
    "agent": {
      "dirs": ["agents"]     ← 缺少闭合大括号
  }
}

当前行为: JSON.parse 抛出 SyntaxError
    → 堆栈追踪打印到终端
    → 用户看到 "Unexpected token } in JSON at position 87"
    → 完全不知道怎么修

需要加固:
    → 捕获 JSON 解析错误
    → 输出: "❌ aiforge.json 第 4 行有语法错误: 缺少 }"
    → 建议: "可以用 npx aiforge validate 检查配置文件"


场景 B: 类型错误
─────────────────────
{
  "structure": {
    "agent": {
      "dirs": "agents"       ← 应该是数组，写成了字符串
    }
  }
}

当前行为: 后续代码假设 dirs 是数组，调用 .forEach() 崩溃
    → TypeError: dirs.forEach is not a function

需要加固:
    → 对 aiforge.json 做 schema 校验
    → 输出: "❌ structure.agent.dirs 应为数组，当前值为字符串 'agents'"
    → 自动修复建议: "你的意思是 'dirs': ['agents'] ?"


场景 C: 引用了不存在的目录
─────────────────────
{
  "structure": {
    "agent": {
      "dirs": ["my-agents"]  ← 仓库中没有这个目录
    }
  }
}

当前行为: 扫描 my-agents/ → 目录不存在 → 空结果
    → 又是静默失败

需要加固:
    → 校验 dirs 中每个路径是否存在
    → 输出: "⚠️ structure.agent.dirs 中的 'my-agents/' 在仓库中不存在"
    → 列出仓库中实际存在的目录，供用户选择
```

### 💥 攻击场景 1.6：DETECTION_PATTERNS 的正则灾难

```
当前设计:
  const DETECTION_PATTERNS = {
    agent: [
      'agents', 'agent', 'our-agents', 'team-agents',
      /^.*-agents?$/,                    ← 这个正则
    ],
  };

攻击:
  目录名: "aaaaaaaaaaaaaaaaaaaaaa-agents"
  → 正则匹配成功，没问题

  但如果正则写错:
  /^(a+)+$/                              ← ReDoS（正则拒绝服务）
  → 对特定输入会指数级回溯
  → Node.js 主线程卡死

  虽然当前的正则 /^.*-agents?$/ 不会触发 ReDoS，
  但这是一个隐含假设:

  隐含假设: "DETECTION_PATTERNS 中的正则是安全的"

  如果未来有人扩展 DETECTION_PATTERNS（或规则包允许自定义正则）:
    → 恶意或写错的正则可能导致 DoS

需要加固:
  1. 对正则设置超时（使用 re2 或手动限时）
  2. 或者: 不允许用户/规则包提供正则，只支持 glob 模式
  3. 内置正则做静态分析确认无回溯风险
```

**这个攻击场景引出一个更深的设计问题：DETECTION_PATTERNS 的正则是否过度设计？**

```
重新评估:
  /^.*-agents?$/ 在实际中匹配的目录名:
    "our-agents"    ← 'our-agents' 字面量已经能匹配
    "team-agents"   ← 'team-agents' 字面量已经能匹配
    "xxx-agents"    ← 真正需要正则的场景

  问题: 有多少团队会用 "xxx-agents" 这种模式？
  → 如果 DETECTION_PATTERNS 的字面量列表足够覆盖常见命名
  → 正则带来的边际收益 < 正则带来的风险和复杂性

  MVP 建议: 只用字面量匹配，不用正则
    const DETECTION_PATTERNS = {
      agent: ['agents', 'agent', 'our-agents', 'team-agents'],
      skill: ['skills', 'skill', 'team-skills', 'capabilities'],
      ...
    };

  非标命名 → 用 aiforge.json 显式声明（这就是 Layer 2 的价值）
```

### 💥 攻击场景 1.7：符号链接模式下的检测混乱

```
用户使用 -l (link) 模式安装后，仓库持久化在本地:
  ~/aicoding-base/
  ├── agents/
  ├── skills/
  └── aiforge.json

  ~/.copilot/agents/ → ~/aicoding-base/agents/  (符号链接)

然后用户在同一个目录再次运行 aiforge:
  $ cd ~/aicoding-base
  $ npx aiforge -p    ← 项目级安装

  问题:
    aiforge 扫描当前目录（即知识仓库自身）
    检测到 agents/, skills/ 等目录
    → 把它们"安装"到项目级目标（.github/, .cursor/ 等）
    → 知识仓库本身被当作了"工作项目"

暴露的脆弱点:
  aiforge 不区分"知识仓库"和"工作项目"
  当用户在知识仓库内运行项目级安装时，会出现荒谬行为

需要加固:
  检测当前目录是否是 aiforge 知识仓库
    if (existsSync('./aiforge.json') || isAiforgeRepo(cwd)) {
      warn("当前目录似乎是一个 aiforge 知识仓库，而非工作项目");
      warn("项目级安装应在你的工作项目目录中运行");
      if (!options.force) process.exit(1);
    }
```

## MVP 必须加固的三道防线

### 防线 1：检测遗漏时的主动报告

```
当 Layer 1 检测结果为空（或部分为空）时:

$ npx aiforge https://gitlab.xxx.com/team/ai-base.git -g

⚠️ 仓库中未发现可识别的配置目录

  仓库根目录包含以下子目录:
    ai-prompts/    ai-rules/    docs/    README.md

  aiforge 能识别的目录名（示例）:
    agents    skills    instructions    mcp-tools

  如果你的配置在上述未识别的目录中，有两种解决方式:

  方式 1: 在仓库中添加 aiforge.json:
    {
      "structure": {
        "agent": { "dirs": ["ai-prompts"] },
        "skill": { "dirs": ["ai-rules"] }
      }
    }

  方式 2: 将目录重命名为 aiforge 约定名称
```

**关键设计点：不是简单地说"没找到"，而是：**

1. 列出实际存在的目录（让用户看到"aiforge 看到了什么"）
2. 列出期望的目录名（让用户理解"aiforge 在找什么"）
3. 给出具体的修复方案（可直接复制的 JSON）

### 防线 2：aiforge.json 的校验与容错

```javascript
function validateAiforgeJson(filePath) {
  // 1. 文件读取
  let raw;
  try {
    raw = fs.readFileSync(filePath, 'utf-8');
  } catch (e) {
    if (e.code === 'ENOENT') return null; // 文件不存在 → 正常，使用 Layer 1
    throw new AiforgeError(`无法读取 ${filePath}: ${e.message}`);
  }

  // 2. JSON 解析（友好的错误提示）
  let json;
  try {
    json = JSON.parse(raw);
  } catch (e) {
    const position = extractPosition(e.message); // "position 87" → 行号列号
    throw new AiforgeError(
      `aiforge.json 语法错误（第 ${position.line} 行）:\n` +
      `  ${e.message}\n\n` +
      `  提示: 可以用 https://jsonlint.com 检查 JSON 格式`
    );
  }

  // 3. Schema 校验
  const errors = [];

  if (json.structure) {
    for (const [type, config] of Object.entries(json.structure)) {
      // 校验语义类型是否已知
      if (!KNOWN_SEMANTIC_TYPES.includes(type)) {
        errors.push(`未知的语义类型 "${type}"，已知类型: ${KNOWN_SEMANTIC_TYPES.join(', ')}`);
      }

      // 校验 dirs 是数组
      if (config.dirs && !Array.isArray(config.dirs)) {
        errors.push(
          `structure.${type}.dirs 应为数组\n` +
          `  当前值: "${config.dirs}"\n` +
          `  修正为: ["${config.dirs}"]`
        );
      }

      // 校验 dirs 中的目录是否存在
      if (Array.isArray(config.dirs)) {
        for (const dir of config.dirs) {
          if (!fs.existsSync(path.join(repoDir, dir))) {
            errors.push(`structure.${type}.dirs 中的 "${dir}" 在仓库中不存在`);
          }
        }
      }
    }
  }

  if (errors.length > 0) {
    throw new AiforgeError(
      `aiforge.json 配置有 ${errors.length} 个问题:\n\n` +
      errors.map((e, i) => `  ${i + 1}. ${e}`).join('\n')
    );
  }

  return json;
}
```

function detectSelfReference(cwd, repoSource) {
  // 场景：用户在知识仓库目录内运行 aiforge -p

  const indicators = [
    path.join(cwd, 'aiforge.json'),
    path.join(cwd, 'aiforge-package.json'),

  const hasAiforgeMarker = indicators.some(f => fs.existsSync(f));

  // 额外检查：当前目录是否就是之前克隆的仓库
  const isClonedRepo = repoSource && 
    path.resolve(cwd) === path.resolve(getClonedRepoPath(repoSource));

  if (hasAiforgeMarker || isClonedRepo) {

    warn('项目级安装 (-p) 应在你的工作项目目录中运行');
    warn('');
    warn('如果你确定要在此安装，请添加 --force');
  
    if (!options.force) {
      process.exit(1);
    }
  }
}

## 混沌工程第一轮的核心收获

```
最重要的发现:
  "静默失败"是 aiforge 设计中最危险的模式。

  它在三个地方出现:
    1. 检测遗漏 → 什么都没装但没报错
    2. aiforge.json 引用了不存在的目录 → 空结果但没报错
    3. overrideDetection 关闭了 Layer 1 但 Layer 2 配置错误 → 空结果

  共同根因:
    代码把"找不到内容"当作"正常的空结果"处理，
    而不是区分"确认没有"和"可能找错了"。

  统一修复原则:
    当最终安装结果为 0 项时，永远不应该只输出 "✅ 完成"
    而应该触发诊断模式，主动输出 "为什么是 0 项" 的分析

    if (installedCount === 0) {
      // 不是:
      log('✅ 安装完成，共 0 项');
    
      // 而是:
      warn('⚠️ 未安装任何文件');
      diagnose(repo, config, detectionResult);
      // 输出: 仓库结构、检测过程、匹配结果、建议
    }
```

**这是混沌工程第一轮最有价值的产出——一个统一的"零结果诊断"机制。**
