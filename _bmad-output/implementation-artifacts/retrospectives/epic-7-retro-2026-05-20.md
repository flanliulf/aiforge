### 1.4 What Was Difficult

Bob (Scrum Master): "这次最大的挑战不是单个工具接入，而是不同工具对同一资源类型的语义不一致。"

Charlie (Senior Dev): "例如 `instructions/` 同时要服务 Claude 的 `CLAUDE.md` 和其他工具的 `AGENTS.md`，这逼出了 `fileFilter`。Windsurf 的 workflows 又逼出了 `semanticWarning`。"

Dana (QA Engineer): "质量挑战也来自基线变动。多个 Story 的规则总量在规划文档、Story AC 和代码现实之间出现过不一致，CR 花了不少力气把口径拉回已批准基线。"

**挑战模式：**

| 挑战 | 具体表现 | 本次处置 |
| ------ | ---------- | ---------- |
| 规则数量口径漂移 | Story 7-1、7-2、7-5、7-6 等出现旧规格数字残留 | 以已批准基线和 Story 明确增量为准，禁止为凑数新增超范围规则 |
| NFR-I5 边界需要解释 | 新工具接入触及 `match-rules`、`execute-install`、`detect-tools` 等受控扩展点 | Story 7-10 产出 NFR-I5 审计，逐项归因边界扩展 |
| 信息性提示容易触发不足或阻断主流程 | Trae Skills、iFlow stale-tool、VS Code migration 等都要求提示但不改变安装语义 | 形成 CR-API-01 / CR-API-02 并同步全局文档 |
| 通用 `.agents/` 与 Antigravity 项目级语义重叠 | `.agents` 既是通用目录，也是 Antigravity 项目检测路径 | 保留现行规则共存行为，并用测试锁定当前约定 |
| 历史 CR TODO 未同步清理 | Epic 6 回顾要求跨 2 Epic TODO 自动升级，但 Epic 7 仍保留 38 open | 作为后续 Critical 准备项继续推进 |

### 1.5 Key Insights

1. **工具生态扩展不是简单加注册表项。** 真正复杂的是资源语义差异：同样叫 agents、skills、instructions，不同工具的含义、路径、启用条件并不一致。
2. **NFR-I5 需要从“绝对零代码改动”升级为“受控扩展点审计”。** Epic 7 证明完全零改动并不现实，但可通过边界归因、数据驱动字段、审计文档保持可控。
3. **规则总量必须绑定已批准基线，不绑定过期规划数字。** Story 7-2 的 `+5 / 24` 裁决和后续多轮 CR 已证明，凑数会比澄清口径更危险。
4. **信息性提示是产品契约，不是日志。** 只要 Story/AC 承诺用户会看到提示，就必须覆盖自动检测、手动指定、异常存在性检查等入口，并保持非阻断边界。
5. **发布型 Story 必须承担文档、测试、审计的收束责任。** Story 7-10 不是“补文档”，而是把 9 个工具接入 Story 的复杂度压缩为可发布证据。
