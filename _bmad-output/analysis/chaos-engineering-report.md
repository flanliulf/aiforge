## 一、执行摘要

本次混沌工程历时完整，覆盖 aiforge 核心系统的 **6 大攻击向量**，共识别 **100+ 个潜在失败场景**，其中 **严重/致命级别 18 个**，**高风险级别 32 个**。

核心结论：**aiforge 的设计方向正确，但在"防御性编程"层面存在系统性缺口**。这些缺口不会阻止 MVP 发布，但若不加固，将导致用户信任危机。

---

## 二、攻击向量汇总

### 向量 #1：双层检测机制

**攻击目标：** 知识仓库的智能检测（Layer 1）+ 显式声明（Layer 2）

| 场景 | 危险度 | 失败模式 |
|------|--------|---------|
| 1.2 检测遗漏（非标准目录名） | **高** | 静默空结果，用户以为工具坏了 |
| 1.5 aiforge.json 语法/类型错误 | **高** | 堆栈追踪或静默失败 |
| 1.7 仓库自引用（在知识仓库内运行 -p） | 中 | 荒谬行为 |
| 1.4 大仓库/Monorepo 扫描深度 | 中 | 嵌套配置被忽略 |
| 1.6 正则 ReDoS 风险 | 低 | 主线程卡死 |

**核心发现：** "静默失败"是最危险的模式。当安装结果为 0 项时，必须触发诊断模式，而非输出"✅ 完成"。

---

### 向量 #2：文件安装机制

**攻击目标：** 复制/符号链接/flatten 三种安装路径

| 场景 | 危险度 | 失败模式 |
|------|--------|---------|
| 2.1 覆盖用户手写文件 | **致命** | 数据丢失，不可逆 |
| 2.2.1 克隆到 /tmp 被系统清理 | 高 | 符号链接全部断裂 |
| 2.2.7 项目级安装 + 符号链接 + Git | **高** | 团队成员出现断链接 |
| 2.3 部分安装失败（半完成状态） | 高 | 系统状态不一致 |
| 2.2.2 Windows 符号链接权限 | 中 | 安装失败，无友好提示 |

**核心发现：** `manifest.json` 是整个系统的状态管理核心，必须在 MVP 中实现。它解决冲突检测、增量更新、健康检查、回滚四个问题。

---

### 向量 #3：认证系统

**攻击目标：** Token 存储、传输、生命周期、多实例管理

| 场景 | 危险度 | 失败模式 |
|------|--------|---------|
| 3.1 Token 出现在进程列表 | **致命** | 共享服务器上任何用户可见 |
| 3.1 Token 出现在 Shell 历史 | **致命** | 永久持久化 |
| 3.1 Token 出现在 Git 配置 | 高 | 仓库存在期间持久化 |
| 3.4 Token 过期无预警 | 高 | 突然失败，用户不知原因 |
| 3.2 权限过大（api/sudo） | 高 | 泄露后影响整个 GitLab 实例 |
| 3.6 企业代理/VPN/自签名证书 | 中 | 连接失败，错误信息不友好 |

**核心发现：** Token 泄露是"系统性风险"，需要分层防御（GIT_ASKPASS + 禁止 --token + 错误脱敏 + 密钥链 + 文件权限 0o600）。预检查比失败后报错更重要。

---

### 向量 #4：语义类型系统

**攻击目标：** 文件类型检测、映射冲突、规则包安全

| 场景 | 危险度 | 失败模式 |
|------|--------|---------|
| 4.3 规则包写入 ~/.ssh/ 等危险路径 | **致命** | 系统文件被覆盖 |
| 4.2 多源映射冲突无优先级 | 高 | 行为不确定 |
| 4.1 无扩展名文件被忽略 | 中 | Dockerfile 等关键文件丢失 |
| 4.5 超大文件内容嗅探 | 高 | 内存溢出 |
| 4.5 循环符号链接 | 高 | 无限循环 |

**核心发现：** 类型检测需要 4 层策略（精确文件名 > 完整扩展名 > 最后扩展名 > 内容嗅探）。规则包安全检查是必需的，不是可选的。

---

### 向量 #5：规则包生态系统

**攻击目标：** 依赖解析、版本管理、恶意包、生态健康

| 场景 | 危险度 | 失败模式 |
|------|--------|---------|
| 5.3 恶意包（Typosquatting/供应链攻击） | **严重** | 窃取数据/植入后门 |
| 5.3 安装脚本攻击 | **严重** | 任意代码执行 |
| 5.3 依赖混淆攻击 | **严重** | 内部包被公开恶意包替换 |
| 5.1 循环依赖 | 中 | 栈溢出 |
| 5.2 语义化版本违反 | 高 | 自动更新破坏项目 |
| 5.5 关键包被废弃 | 高 | 生态单点故障 |

**核心发现：** MVP 阶段规则包生态系统应保持简单（Git 仓库直接安装），不引入中心化注册表。安全扫描和分层信任模型是生态成熟后的必要投入。

---

### 向量 #6：更新机制的安全边界

**攻击目标：** 增量更新、冲突合并、回滚机制

| 场景 | 危险度 | 失败模式 |
|------|--------|---------|
| 6.1.1 差异伪造攻击 | **严重** | 恶意规则被注入 |
| 6.1.2 版本降级攻击 | **严重** | 安全修复被撤销 |
| 6.3.1 快照污染 | **严重** | 回滚到恶意状态 |
| 6.3.2 回滚不完整 | **严重** | 文件版本不一致 |
| 6.3.4 回滚逃逸 | **严重** | 项目外恶意载荷残留 |
| 6.2.3 TOCTOU 竞态条件 | 高 | 文件损坏/数据丢失 |

**核心发现：** 更新机制的三个设计原则：增量更新"信结果不信过程"（哈希校验高于 diff 信任）；冲突合并"结构高于文本，安全高于便利"；回滚"前进到已验证的旧状态，而非简单撤销"。

## 三、核心洞察提炼

### 洞察 1：静默失败是最危险的设计模式

贯穿 6 个攻击向量的共同根因：系统把"找不到内容"当作"正常的空结果"处理。

**统一修复原则：**
> 当最终安装结果为 0 项时，永远不应该只输出"✅ 完成"，而应该触发诊断模式，主动输出"为什么是 0 项"的分析。

---

### 洞察 2：分层防御优于单点防御

Token 安全、文件安装、类型检测都揭示了同一个模式：单一防线必然被突破，分层防御才能提供真正的保护。

**Token 安全的五层防线：**
1. GIT_ASKPASS（避免进程列表暴露）
2. 禁止 --token 参数（避免历史记录）
3. 错误信息脱敏（避免日志泄露）
4. 系统密钥链存储（避免文件泄露）
5. 文件权限 0o600（最后防线）

---

### 洞察 3：预检查比失败后报错更有价值

**传统流程（被动）：** 开始操作 → 失败 → 报错 → 用户修复 → 重试
**预检查流程（主动）：** 预检查 → 发现问题 → 精确提示 → 用户修复 → 开始操作

投入：3-5 个 API 调用（几秒）
回报：避免 90% 的失败和困惑

---

### 洞察 4：manifest.json 是系统的状态管理核心

它不只是"记录装了什么"，而是整个系统的基础设施：
- **冲突检测：** 目标文件是 aiforge 的还是用户的？
- **增量更新：** 上次安装了什么？这次有什么变化？
- **健康检查：** 符号链接是否还有效？
- **回滚/清理：** 需要撤销时删哪些文件？

---

### 洞察 5：终端输出是"对话"而非"日志"

每一行输出都应该回答用户脑中的一个问题。错误信息的三段式结构：
1. **什么坏了**（用户语言描述问题）
2. **为什么**（简短解释原因）
3. **怎么修**（可直接复制执行的命令）

---

### 洞察 6：MVP 阶段应主动降低复杂度

混沌工程揭示了多个"过度设计"的风险点：
- 规则包生态系统 → MVP 阶段只用 Git 仓库直接安装，不引入注册表
- 符号链接模式 → 项目级安装禁用 -l，只有全局安装支持
- flatten 模式 → 只做路径转换，不做内容转换
- DETECTION_PATTERNS 正则 → 只用字面量匹配，不用正则

## 七、关键架构决策

基于混沌工程发现，以下架构决策需要在实现前确认：

1. **manifest.json 位置：** `~/.aiforge/manifest.json`（全局）vs 每个工具目录下（分散）
   - **推荐：** 全局单一 manifest，按工具和安装范围分组

2. **克隆仓库位置：** `~/.aiforge/repos/<repo-hash>/`
   - **决策：** 固定位置，不使用 /tmp

3. **符号链接策略：** 全局安装支持 -l，项目级安装禁用 -l
   - **决策：** 已确认，防止 Git 追踪问题

4. **Token 存储：** keytar 系统密钥链（主）+ 环境变量（降级）
   - **决策：** 禁止 --token 命令行参数，禁止明文配置文件

5. **规则包生态 MVP 范围：** 只支持 Git 仓库直接安装，不引入注册表
   - **决策：** 降低 MVP 复杂度，生态建设是阶段 3 的工作
