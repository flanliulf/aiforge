# Product Brief: ai-forge

## Executive Summary

aiforge 是一个通过 `npx` 运行的命令行工具，解决 AI 编码工具配置碎片化的核心痛点。在 AI 编码辅助工具（GitHub Copilot、Claude Code、Cursor、Windsurf 等）爆发式增长的当下，每个工具对 Instructions、Skills、Agents、MCP Tools 等配置资源有着完全不同的目录约定和格式要求。aiforge 让团队和个人只需维护一份知识仓库，一条命令即可将配置自动安装到所有 AI 工具中。

aiforge 的设计哲学是"标准适配器"而非"标准制定者"——知识跨平台可移植，安装实现平台特化。工具本身发布在 npm 公网，不含任何敏感信息；知识仓库托管在私有 Git，认证完全在用户侧完成。这种工具与知识的完全解耦，使得任何组织都可以创建自己的知识仓库并通过 aiforge 分发。

MVP 阶段聚焦公司内部开发者和平台维护者的核心场景，同时按社区级通用工具的标准进行架构设计，为后续开源推广奠定基础。

---

## Core Vision

### Problem Statement

随着 AI 编码辅助工具的爆发式增长，企业和团队面临一个日益严重的配置管理困境：每个 AI 工具（Copilot、Claude Code、Cursor、VS Code、Windsurf）都有独立的配置目录约定、文件格式要求和安装路径。团队精心编写的 AI 编码规范（Instructions、Skills、Agents、MCP Tools）需要手动适配到每个工具中，且缺乏统一的分发和更新机制。

当前的 Shell 脚本方案存在六大痛点：
1. 只支持符号链接，要求仓库持久存在于本地特定路径
2. 只支持单一工具，硬编码了特定工具的路径
3. 跨平台差异大，Windows 上需要 WSL
4. 无自动检测，开发者需要手动选择安装目标
5. 无增量更新判断，每次全量覆盖
6. 全局安装和项目安装是两个不同的脚本

### Problem Impact

- **开发者层面：** 新员工入职需要阅读 Shell 脚本才能完成 AI 工具配置，使用 Cursor/Windsurf 等新兴工具的开发者完全无法享受团队统一配置
- **平台团队层面：** 每新增一个 AI 工具支持，就需要编写新的安装脚本，维护成本线性增长
- **组织层面：** AI 编码规范的采用率受限于安装体验，团队知识资产无法高效流通

### Why Existing Solutions Fall Short

| 现有方案 | 局限性 |
|---------|--------|
| Shell 脚本（setup.sh） | 单工具绑定、跨平台差异、无自动检测、无增量更新 |
| 手动复制文件 | 易出错、无法追踪版本、更新成本高 |
| 各工具原生配置同步 | 各自为政，无法跨工具统一管理 |
| dotfiles 管理工具（stow/chezmoi） | 通用文件管理，不理解 AI 工具的语义类型和目录约定 |

核心差距：没有一个工具能够理解"AI 编码配置"的语义（agent、skill、instruction、mcp-tool），并将其自动映射到不同 AI 工具各自的目录约定中。

### Proposed Solution

aiforge 是一个通用的、跨平台的 AI 编码配置安装 CLI，核心能力包括：

- **智能检测：** 自动扫描用户环境中安装的 AI 工具，无需手动指定
- **语义映射：** 理解四类资源（agents/skills/instructions/mcp-tools）的语义，按每个工具的约定自动安装到正确位置
- **双模安装：** 支持复制模式（项目级快照）和符号链接模式（全局长期使用，git pull 自动生效）
- **多源认证：** 四层认证优先级（CLI 参数 > 环境变量 > 配置文件 > 系统凭据），适配开发、CI/CD、新用户等不同执行上下文
- **配置驱动扩展：** 新增 AI 工具支持只需添加配置规则，不改引擎代码

用户体验目标：一条命令完成所有 AI 工具的配置安装，终端输出是"对话"而非"日志"。

### Key Differentiators

1. **工具与知识完全解耦** — aiforge（npm 公网）不含任何仓库 URL、Token 或公司信息；知识仓库（私有 Git）独立迭代。任何组织都可以创建自己的知识仓库并通过 aiforge 分发，无需 fork 工具本身。

2. **语义适配层** — 不是简单的文件复制工具，而是理解 AI 编码配置语义的"翻译器"。用户操作语义层面的概念（"安装 skills"），aiforge 翻译成平台和工具特定的物理操作。

3. **零配置到完全定制的渐进式体验** — 无 aiforge.json 时智能检测自动工作；有 aiforge.json 时支持完全自定义的结构映射和安装规则。降低采用门槛的同时满足高级需求。

4. **时机优势** — AI 编码工具正处于爆发期，Copilot/Claude Code/Cursor/Windsurf 等工具各自为政，配置碎片化问题只会越来越严重。aiforge 在这个窗口期提供统一的配置分发方案，有机会成为事实标准。

## Target Users

### Primary Users

#### 用户画像 1：小明 — 公司中高级开发者

- **背景：** 3-5 年经验的后端/全栈开发者，日常使用 VS Code + GitHub Copilot 作为主力 AI 编码工具，同时会用 Claude Code 处理复杂重构任务。偶尔尝试 Trae、Codex 或 Gemini CLI 等新兴工具。熟悉命令行和 Git，但不想在"配置管理"上花时间。
- **核心动机：** 希望 AI 工具能按照团队规范工作，但不想手动折腾每个工具的配置目录。
- **当前痛点：**
  - 入职时被告知"跑一下 setup.sh"，但脚本只覆盖了 Copilot，自己常用的 Claude Code 需要手动复制
  - 团队更新了 AI 编码规范后，不知道自己本地的配置是不是最新的
  - 尝试新工具（如 Trae）时，完全没有团队配置可用，只能裸奔
- **成功愿景：** 一条命令搞定所有工具的配置，换新工具时再跑一次就行，不用关心文件放在哪里。
- **信任顾虑：** "这个工具会不会覆盖我自己手写的配置？" — 需要 `--dry-run` 预览和冲突备份机制来建立信任。

#### 用户画像 2：小李 — 初级开发者（新员工）

- **背景：** 刚入职的应届生或 1 年经验的开发者，对命令行基本操作没问题，但不熟悉符号链接、环境变量等概念。公司给配了 VS Code + Copilot，可能还没听说过 Claude Code。
- **核心动机：** 快速融入团队的开发规范，不想因为工具配置问题显得"不专业"。
- **当前痛点：**
  - 入职文档里的 setup.sh 步骤看不太懂，不确定自己执行对了没有
  - 遇到认证问题（SSH Key 没配好）时不知道怎么排查
  - 不敢随便跑脚本，怕搞坏环境
- **成功愿景：** 按照入职文档跑一条命令，看到清晰的成功提示，就知道配好了。遇到问题时，工具能告诉自己"什么坏了、为什么、怎么修"。
- **关键需求：** 交互式引导（`aiforge init`）、三段式错误提示、零配置路径（系统凭据自动生效）。

### Secondary Users

#### 用户画像 3：chunxiao — 平台团队维护者

- **背景：** 负责 `aicoding-base` 知识仓库的维护和 aiforge 工具的开发。需要确保新增的 skill/agent 能正确安装到各个 AI 工具中，同时推动全公司采用统一配置。
- **核心动机：** 降低维护成本，提高团队 AI 编码规范的采用率。
- **当前痛点：**
  - 每新增一个 AI 工具支持，就要改 Shell 脚本，测试多个平台
  - 推动采用时，开发者的反馈是"太麻烦了"或"怕搞坏配置"
  - 无法确认谁装了、装的是哪个版本、有没有装成功
- **成功愿景：** 新增工具支持只需在配置表里加几行规则；`--dry-run` 快速验证安装计划；团队采用率从被动推广变为"一条命令，没有理由不用"。
- **关键需求：** `--dry-run` 预览模式、按工具分组的树形结果展示、配置驱动的可扩展性、零结果诊断模式。

#### 潜在用户：外部开发者 / 社区

- **背景：** 其他公司或个人开发者，有自己的 AI 编码知识仓库，希望用通用工具分发配置。
- **MVP 阶段定位：** 不主动服务，但架构设计不绑定特定公司，确保未来开源后可直接使用。
- **关键约束：** npm 包不含任何公司信息、仓库 URL 或 Token。

### User Journey

#### 核心旅程：小明的首次使用

```
发现 → 信任建立 → 安装 → Aha Moment → 日常使用
```

| 阶段 | 触发 | 行为 | 情绪 | 产品响应 |
|------|------|------|------|---------|
| **发现** | 团队通知"用 aiforge 替代 setup.sh" | 阅读简短的使用说明 | 半信半疑 | — |

| **安装** | 运行 `npx aiforge -g -l` | 观察阶段式进度输出 | 期待 | spinner + 实时状态 + 按工具分组的结果树 |

| **日常更新** | 团队通知"aicoding-base 更新了" | 运行 `npx aiforge update` | 轻松 | git pull 自动生效（符号链接模式） |

#### 关键旅程：小李的入职配置

```
入职文档 → aiforge init → 认证配置 → 一键安装 → 确认生效
```

| 阶段 | 行为 | 可能的阻塞 | 产品响应 |
|------|------|-----------|---------|

| **认证失败** | 系统凭据不可用 | 不知道什么是 Token | 三段式提示：什么坏了 / 为什么 / 怎么修（含 Token 获取链接） |
| **安装成功** | `npx aiforge -g -l` | — | 清晰的成功输出，新手也能确认"装好了" |

## MVP Scope

### Core Features

#### 安装核心（必须完整可用）

| 功能 | 描述 | 对应 PRD |
|------|------|---------|
| **Git 仓库克隆** | 支持 HTTPS 和 SSH 协议，浅克隆（`--depth 1`） | FR-001~FR-005 |
| **四工具支持** | GitHub Copilot、Claude Code、Cursor、VS Code | FR-013~FR-015, FR-043 |
| **全局/项目安装** | `-g` 全局目录，默认项目目录 | FR-016, FR-017 |
| **四类资源安装** | agents、skills、instructions、mcp-tools | FR-018 |
| **自动工具检测** | 扫描标志性文件/目录判断已安装的 AI 工具 | FR-013 |
| **复制模式** | 文件从源仓库复制到目标目录（默认） | FR-019 |
| **符号链接模式** | `-l` 持久化仓库 + 创建符号链接（仅全局安装） | FR-020, FR-021 |
| **安装规则映射表** | 内置静态规则，配置驱动，新增工具只改配置 | FR-023, FR-043~FR-044 |

#### 认证系统（基础可用）

| 功能 | 描述 | 对应 PRD |
|------|------|---------|
| **SSH Key 认证** | URL 自动转换为 SSH 协议 | FR-006 |
| **Token 认证** | 支持 CLI `--token`、环境变量（`AIFORGE_TOKEN`/`GITLAB_TOKEN`）、配置文件 | FR-007, FR-008 |
| **系统凭据降级** | 无显式认证时，由 git 调用系统凭据管理器 | FR-009 |
| **配置文件** | `~/.aiforge/config.json` 保存默认仓库和认证方式 | FR-040 |
| **错误信息脱敏** | 日志中 Token 显示为 `glpat-ab****xy` 格式 | FR-012 |

#### 用户体验（Aha Moment 的保障）

| 功能 | 描述 | 对应 PRD |
|------|------|---------|
| **交互式初始化** | `aiforge init` 引导首次配置（仓库 URL、认证方式、连接验证） | FR-033 |
| **预览模式** | `--dry-run` 展示安装计划，不写入文件 | FR-034 |
| **安装结果汇总** | 按工具分组的树形输出 + 统计数字 | FR-036 |
| **阶段式进度** | spinner + 实时状态更新 | 头脑风暴 UX #1 |
| **三段式错误提示** | 什么坏了 / 为什么 / 怎么修 | 头脑风暴 UX #3 |

#### 可靠性基础设施（混沌工程 P0）

| 功能 | 描述 | 来源 |
|------|------|------|
| **manifest.json** | 安装状态管理核心——记录已安装文件、hash、时间戳 | 混沌工程 #2 |
| **文件冲突处理** | 基于 manifest 的冲突检测 + 备份（`.bak`）+ `--force` 覆盖 | FR-025~FR-029 |
| **零结果诊断** | 未产生任何可处理项时触发诊断输出（扫描了哪些目录、匹配了哪些模式）；「全部跳过」场景不在此列，视为成功路径 | 混沌工程 #1 |
| **克隆路径固定** | 符号链接模式克隆到 `~/.aiforge/repos/` 而非 `/tmp` | 混沌工程 #2 |
| **项目级禁用 `-l`** | 项目安装不支持符号链接模式，防止团队断链接 | 混沌工程 #2 |

### Out of Scope for MVP

| 明确不做 | 原因 | 计划阶段 |
|---------|------|---------|
| `aiforge update` 命令 | 符号链接模式下 git pull 已可手动更新 | M2 |
| `aiforge list` 命令 | 非核心路径，文档可替代 | M2 |
| Windsurf / Codex CLI / Gemini CLI / Trae 支持 | 用户量少，优先覆盖主流四工具 | M2 |
| `aiforge uninstall` 命令 | manifest.json 已为此奠基，但 MVP 不实现 | M3 |
| `aiforge.json` 自定义清单 | MVP 使用内置规则映射表，够用 | M3 |
| 格式转换（Skills → .mdc） | MVP 只做路径映射，不做内容转换 | M3 |
| GIT_ASKPASS 安全机制 | MVP 基础认证可用，M2 安全加固 | M2 |
| keytar 密钥链存储 | MVP 用配置文件 + 环境变量，M2 增强 | M2 |
| Windows 支持 | 公司以 macOS 为主，Windows 需额外适配 | M2 |
| 规则包生态系统 | 远期愿景，MVP 只支持单一 Git 仓库 | P3 |
| 语义类型系统 | 远期愿景，MVP 用目录名匹配 | P3 |
| 增量更新（哈希比对） | manifest.json 记录 hash，但 MVP 不做增量判断 | M3 |

### MVP Success Criteria

MVP 发布的 Go/No-Go 决策基于以下门禁：

| 门禁 | 标准 | 验证方式 |
|------|------|---------|
